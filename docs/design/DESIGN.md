## Accepted amendments from the stress test

The adversarial review (STRESS-TEST.md) produced these binding changes to the proposal below:

1. **Primary persona = local-first / desktop / CLI / agent-memory apps** (single process by nature). FastAPI/GraphQL demoted from flagship to integration; `workers=1` is a documented constraint until a multi-process story exists.
2. **Define the concurrency contract before v0.1 code** (single-owner model with commit-time write barrier, or session-scoped copies — one-way door).
3. **Start on SQLite-as-blob-store** behind a 3-method storage protocol; the custom append-only log becomes a later optimization gated on profiling.
4. **RDF/pyoxigraph moves from core to an extension package**; core caps at two sidecar persistence technologies. (Under re-evaluation: native triple indexes on our own bitmap machinery — round-2 research.)
5. **Memory honesty**: ~600 B per live registered object all-in; positioning envelope ~1–5M live objects. Compact array-backed boot index + persisted index checkpoints required (startup must be O(tail), not O(write history)).
6. **Durability**: per-record checksums (recovery checksum-driven, not length-driven); `F_FULLFSYNC` on macOS; interval-fsync group commit as documented default; sidecar-ahead-of-core detection forces index rebuild.
7. **Schema evolution**: rename heuristic OFF by default; offline migrate/verify command in v0.2; defined behavior for old-code-opens-newer-store.
8. **Rename the package before any public artifact** (pyrsistent has 41.7M downloads/month).
   ✅ Resolved 2026-06-10: renamed to **datacrystal** ([NAME.md](NAME.md)); PyPI names
   `datacrystal` + `data-crystal` still to be reserved before anything goes public.
9. **Timeline honesty**: v0.1 ≈ 2–3 months (entity decorator, OID registry, msgspec records over SQLite-blob storage, Lazy[T], pyroaring queries); full v1.0 scope ≈ 24 months.

## Feasibility verdict

YES — an EclipseStore-for-Python is feasible, and ZODB is 25 years of production proof that the core concept (transparent object-graph persistence, ghosts/lazy refs, append-only log storage, per-object records) works in CPython: Plone/FBI/NASA deployments, 44M+ object RelStorage installs, and the whole stack still shipping releases in 2026 (ZODB 6.3, persistent 6.7). The JVM heap is not what makes EclipseStore work — what it actually relies on maps cleanly, and in several places CPython is BETTER. What replaces what: (1) Java's WeakReference dual-hash-table swizzle registry + ReferenceQueue draining → weakref.WeakValueDictionary keyed by OID; refcounting fires callbacks at the exact moment the last strong ref dies, which eliminates the entire 'cleared-but-unreaped window' defense complex documented in EclipseStore's 440-line GC.md (their hardest correctness problem simply disappears). (2) Unsafe.allocateInstance constructor bypass → object.__new__(cls) + direct __dict__/__slots__ writes (and pydantic model_construct) — same no-code-execution-on-load security posture. (3) Java reflection type-handler generation → dataclasses.fields()/pydantic introspection, which is richer and gives typed schema for free. (4) Lazy<T> wrapper → explicit Lazy[T] plus a benchmarked-novel 'class-swap ghost' (subclass overrides __getattribute__, loads state, assigns __class__ back) with ZERO steady-state attribute overhead — beating even ZODB's C extension (1.6x reads) without shipping C. (5) JVM heap-quota lazy clearing → psutil RSS budget with the same age-weighted formula. Where Python is honestly WORSE: ~135 B per slots-dataclass object vs ~32-40 B in Java (3.5-4x RAM; 10M objects ≈ 1.4-1.7 GB), object-graph scans at 18-26 ns/object with no parallelStream escape (measured 2.5x single-thread penalty on free-threaded 3.14t — not a way out through ~2027), and 464 ms cyclic-GC pauses at 10M live objects (fixable with gc.freeze). The architecture must therefore lean into Python's strengths — refcounting determinism, runtime dynamism (__class__ assignment, descriptors, Annotated metadata), and the scientific stack: columnar Arrow/numpy mirrors answer the scan problem at 0.76 ns/row, which actually BEATS single-threaded Java streams. The one thing not to port is the multi-channel apparatus — it exists for JVM multi-core write parallelism the GIL can't use.

## Core architecture

ENGINE (single process, single writer, embedded). Storage: port EclipseStore's append-only design at channel-count=1 — data files as pure concatenations of entity records with the 24-byte header [8B length][8B typeId][8B objectId] (struct '<qqq'), 8MB rollover, plus the tiny fixed-size binary transaction log (FILE_CREATION/STORE/TRANSFER/TRUNCATION/DELETION entries recording file-length checkpoints). Recovery is EclipseStore's ~40-line discipline: validate file lengths against the log, truncate the torn tail, log-before-any-destructive-action. Keep channel index in filenames so multi-channel stays format-compatible later. One deliberate improvement: configurable fsync (per-commit / interval / never) — EclipseStore silently never calls force() and is consistent-but-not-durable-to-device; we make that an explicit policy with group-commit batching for throughput. Boot = one reverse-order scan rebuilding dict[int, EntityRef(file, offset, length, tid)] newest-version-wins; reads via mmap (Python objects are the cache — skip EclipseStore's off-heap entity cache entirely). OIDs/registry: EclipseStore's partitioned 64-bit ID space verbatim (TIDs < 1e18, OIDs in (1e18, 9e18), constant-IDs above for enum members/sentinels so any 8-byte slot is self-classifying); OID→object identity map is a WeakValueDictionary; reverse lookup by stamping __pyr_oid__ on instances (Python can, Java couldn't), id()-keyed dict + weakref.finalize fallback for closed __slots__ classes. Object model: slots dataclasses with weakref_slot=True as the canonical stored form (~135 B/obj measured); pydantic BaseModel explicitly NOT the storage model (588 B/obj, slowest encode) — it lives at the API edge. Payload codec: msgspec msgpack inside the record body (93 ns/obj, 12-14x faster than pickle-p5, validating decode, structurally incapable of code execution), references swizzled to OIDs; PEP 574-style out-of-band buffers for numpy/Arrow blob fields; pickle nowhere in the canonical format. Lazy refs: explicit Lazy[T] (.get()/.peek()/.clear()) as the documented API — EclipseStore's own anti-proxy argument — plus auto-generated class-swap ghosts for intra-graph references; a LazyReferenceManager daemon thread (1s interval, 1ms budget, weakly-held entries) clears by timeout + RSS quota (psutil, sampled once per cycle, EclipseStore's age-weight formula), optionally triggering gc.collect() so cyclic subgraphs actually die; gc.freeze() after load/commit batches kills the 464 ms pause. Dirty tracking — fix EclipseStore's worst footgun (silent loss of in-place child mutations under the lazy storer): tri-state class swap Ghost→Clean→Dirty, one ~150 ns __setattr__ hit on FIRST write per object per transaction, zero after; msgspec-fingerprint diff at commit for frozen/third-party types; ship PersistentList/PersistentDict as default container field types (ZODB's _p_changed lesson). Transactions: buffer-until-commit storer with EclipseStore semantics (argument always rewritten, registered children skipped; eager storer opt-in), single global writer lock, commit = append + log entry + counter + fsync policy; the commit record also carries index deltas (see search pillar). Schema: append-only human-readable type dictionary file (JSON-lines .ptd analog) — TIDs are the only type metadata in binary records; new TID per structural version; legacy type mapping ported verbatim (startup diff, compiled per-legacy-TID converters, auto add/remove/reorder/widen, difflib rename heuristic) with Annotated markers as the primary evolution API (Annotated[str, RenamedFrom('name')], default_factory for added fields) and a refactorings TOML escape hatch for class renames; restrict the persistable universe to dataclasses/attrs/msgspec.Struct + builtins + bytes/numpy/Arrow buffers — the type dictionary becomes a hard security whitelist. Disk GC: do NOT port the incremental tri-color mark&sweep (~2000 lines, exists only to avoid pauses on huge JVM heaps); explicit/idle stop-the-world mark&sweep using GC.md as the spec (snapshot registry keys as both mark seed and sweep keep-alive — exact in Python), plus file dissolving by live-ratio (0.75 threshold, 1MB/8MB bounds) as an on-demand/background task.

## Query story

The GigaMap lesson is the heart of question 4: EclipseStore's baseline answer to 'complex queries' is no query engine at all — Java Streams over the in-heap graph, hand-rolled HashMap/TreeMap indexes, and an explicit warning never to parallelize across Lazy boundaries; the 'microseconds' claims are marketing comparing pointer-chasing to ORM round-trips. At scale even the JVM gives up on scanning: GigaMap exists because iterating a billion heap objects is too slow, so it keeps entities in a 3-level lazy segment tree (256/leaf) and answers queries from off-heap compressed bitmap indexes, deserializing only segments containing hits. In Python this lesson bites ~100x earlier: measured scans are 18-26 ns/object (0.2-0.5 s per 10M, single-threaded, GIL) and 10M resident objects cost 1.4-6 GB — so unindexed full scans are a warned fallback above ~10^5-10^6 entities, never the strategy. pyrsistance's answer is two-tier. Tier 1, selective queries → EntityMap (the GigaMap analog): sequential int64 entity ids; segmented lazy pages (256-4096 entities, dirty-flag partial persistence); indexers derived from Annotated field markers; a Condition AST built by operator overloading ((Person.city == 'Berlin') & (Person.age >= 18)); bitmaps via pyroaring instead of EclipseStore's hand-rolled off-heap hierarchy — CRoaring gives compressed, GC-invisible AND/OR/NOT measured at 306 µs across a 10M-id space, i.e. the entire query kernel for free; copy the predicate-over-distinct-keys trick (contains/startswith iterate O(cardinality) index keys, never entities); identity indexes for update/remove, unique constraints validated on write; results = bitmap of ids → hydrate only touched pages through the registry. Tier 2, analytics/aggregations → columnar Arrow mirrors, the thing GigaMap CANNOT do and Python's genuine edge: per-entity-class ColumnarProjection (mandatory uint64 __oid__ column + typed columns of indexed scalar fields), maintained inside the commit (insert = buffered RecordBatch flush, delete/update = roaring tombstone bit, merge-on-read à la Delta/Lance deletion vectors, threshold-driven compaction, atomic snapshot swap = free snapshot isolation), registered zero-copy in DuckDB (filter/projection pushdown, 11x-2900x over pandas in DuckDB's own benchmarks) with polars LazyFrame as a second zero-copy surface via __arrow_c_stream__. Measured: numpy/Arrow masks at 0.76-2.3 ns/row — 14-109x faster than Python object loops and faster than single-threaded Java streams; and columnar kernels release the GIL, which is a better parallelism story than free-threading through at least 2027. Hard rule learned from DuckDB issue #2450: never expose object-dtype columns — typed extraction happens once per commit, not per query. When each applies: bitmap index for point/range/boolean predicates returning live objects; Arrow+DuckDB/polars for group-by/joins/aggregations and data-science handoff; raw scan only below ~100K entities. Hits always swizzle back via the __oid__ column through the identity map — query in columns, touch objects only for final results.

## Search, vector, RDF

All three pillars baked in, following EclipseStore's one good structural idea (Lucene/JVector as sibling index plugins on the same collection, declared by field annotation) but NOT its transactional naivety. Chosen engines — defaults, not menus: FULL-TEXT = SQLite FTS5 sidecar ('indexes.db' colocated in the store directory, contentless/external-content tables keyed by OID, BM25, trigram tokenizer) — the only 2026 candidate with truly transactional per-row incremental updates and zero added dependency (stdlib sqlite3; probe FTS5 at store-open with apsw/pysqlite3-binary fallback); tantivy-py (0.26.0, Apr 2026, ft wheels) as the high-performance opt-in extra with debounced group commits. VECTOR = usearch (v2.25.3 May 2026, Apache-2.0, single-file save/load + mmap view, true incremental add/remove, numpy end-to-end, 3.13t resolved) — one .usearch file per @Vector field; sqlite-vec is the documented 'unified transactional sidecar' alternative under ~500K vectors and the watch item: if its DiskANN/IVF (0.1.10-alpha) stabilizes, one SQLite sidecar could absorb the vector default. Genuinely close call: usearch vs LanceDB — Lance also buys versioned columnar persistence + IVF_PQ, but embeds an entire second database engine; usearch wins v1 on footprint. RDF = pyoxigraph (0.5.8 Apr 2026, MIT/Apache, RocksDB store with documented snapshot isolation and atomic extend(), full SPARQL 1.1, cp313t/cp314t wheels) as the triple index, with an rdflib Store facade (oxrdflib is the ~600-LOC reference) so pyrsistance objects are SPARQL-queryable through the standard Python RDF API; triples projected from entities via Annotated markers (net-new — nothing in EclipseStore does RDF). Hard disqualifications, for the record: DuckDB FTS (full PRAGMA rebuild, no incremental), DuckDB VSS (experimental persistence, WAL-recovery corruption risk), Kuzu (repo archived Oct 2025), CozoDB (dead since 2023), whoosh (pure-Python, 2024), hnswlib (dormant, no wheels). COMMIT-PIPELINE INTEGRATION — the design rule that makes 'baked in' honest: indexes are rebuildable derived data, never distributed-transaction participants. During a store transaction, index deltas (upsert/delete by OID) accumulate beside dirty objects and are persisted IN the commit record (the transaction-log analog) — the replayable write-ahead source. Post-commit, each index applies deltas idempotently against a last_applied_commit_id watermark: the SQLite sidecar synchronously in ONE sidecar transaction that also bumps its watermark table (crash-safe, read-your-writes — leaning on real transactionality Java never had); usearch applies in-memory immediately, snapshots via atomic rename at checkpoint/close, replays the log tail on open; pyoxigraph applies via atomic extend()/remove() with the watermark stored as a quad in a meta named-graph. On open: replay from each watermark; full rebuild from the object graph is always a valid recovery path. This is the direct fix for ZODB's two index failure modes — bolt-on ZCatalog ConflictError hot spots and reindex-by-convention.

## Ecosystem integration (dataclasses, strawberry, FastAPI, Arrow)

One class definition, everything derived — verified end-to-end against strawberry 0.316.0 source and empirical tests. The keystone is being dataclass-first, not pydantic-first: @strawberry.type is literally dataclasses.dataclass + a sidecar __strawberry_definition__ (the same class object passes through unchanged), pydantic v2 TypeAdapter validates/serializes dataclasses including Annotated constraints, and FastAPI accepts dataclasses as response models — a plain dataclass is the only representation all three consume natively. So: @pyr.entity applies @dataclass(slots=True, weakref_slot=True) if absent, instruments attributes AFTER dataclass processing (SQLAlchemy-style; verified invisible to strawberry, which reads via plain getattr and harvests __dataclass_fields__), and registers field metadata from Annotated markers — pyr.Id, pyr.Lazy, pyr.Index, pyr.FullText, pyr.Vector, pyr.Transient (mapped to strawberry.Private) — the verified pattern where one Annotated annotation simultaneously carries persistence, GraphQL, and pydantic semantics and each library ignores the others' markers. Pydantic: edge layer only — cached TypeAdapter per entity for REST/JSON and OPTIONAL validate-on-load (a genuine improvement: EclipseStore's constructor bypass can never re-check invariants; Python can, behind a flag); BaseModel-defined entities get a second-class path via a small generated-dataclass shim (reimplemented, not depending on strawberry.experimental). Strawberry/GraphQL — the centerpiece mechanism, empirically verified: StrawberryConfig(default_resolver=store_resolver) where the resolver detects unloaded Lazy/ghost refs and returns DataLoader futures — graphql-core awaits them and sibling refs batch into ONE storage load — giving transparent, batched lazy-reference resolution across the whole schema with zero per-field resolver code; int64 OIDs surface as relay GlobalID('Type:oid') with one registry-level resolve_nodes against the store; input types auto-generated from the same metadata. Vendor strawberry's 300-line dependency-free DataLoader into the core (per-request loaders with cache=False, primed from the identity map) so the persistence layer has no strawberry dependency. FastAPI: pyr.fastapi.router(store) wraps GraphQLRouter with context_getter (a real Depends-chain dependency) injecting store + request-scoped loaders; store lifecycle via lifespan; configuration via pydantic-settings reusing EclipseStore's flat key names (storage-directory, …). Critical async rule: the persistence core stays synchronous (EclipseStore-style blocking Lazy.get()), but getattr-triggered disk reads must never run inline on the event loop — they route through the default_resolver awaitable hook or anyio.to_thread; this asyncio-native facade is ZODB's weakest flank (issue #311 open since forever) and a launch differentiator. pandas/pyarrow: no 'integration' needed — the columnar mirrors ARE Arrow, so store.arrow(Person)/.to_pandas()/.to_polars() are zero-copy views and Parquet/Lance export is one call; this is the answer to 'worth considering given the scientific stack' — it is the load-bearing wall, not an add-on.

## Positioning

One-line pitch: 'Your Python objects ARE the database — typed, crash-safe, pickle-free object-graph persistence with built-in SQL, full-text, vector, and SPARQL queries, embedded in your process like SQLite.' Why it deserves to exist: the transparent object-graph niche in Python is dormant, not disproven — ZODB (205K downloads/month, still releasing in 2026) proves demand and feasibility while diskcache (39.3M/month) proves Python developers eagerly adopt embedded serverless persistence when ergonomics are right; everything in between is key-value/document (tinydb, sqlitedict, shelve) with no graph, no identity, no transactions-over-objects. vs ZODB: every documented ZODB failure is a designed-in fix here — opaque pickle that breaks on class renames → msgspec records + versioned type dictionary + legacy type mapping (and no load-time code execution, ZODB's standing CVE-shaped flaw); manual _p_changed → automatic class-swap dirty tracking; zero built-in search (dead Newt DB proves half-measures don't stick) → FTS/vector/RDF in the commit pipeline; no asyncio/typing story → async facade + types-as-schema from day one. vs sqlite/ORMs: no impedance mismatch, no mapping layer, identity-preserving cyclic graphs, schema = your dataclasses. vs duckdb: DuckDB is an analytics engine over tables, can't be your application's object model (and its FTS/VSS indexes are rebuild-only/experimental) — pyrsistance embeds DuckDB as its analytics surface rather than competing. vs lancedb: tables-and-vectors, not graphs; same relationship — potential durable columnar tier, not a rival. The unfair advantages no JVM product can copy: zero-copy Arrow/pandas/polars/DuckDB views of your live object graph, and an end-to-end typed story from dataclass to GraphQL schema. NAME RISK, flagged clearly: 'pyrsistance' is one letter-cluster from 'pyrsistent' (Tobias Gustafsson's widely-used immutable-collections library, millions of downloads/month) — guaranteed confusion in search/conversation ('did you mean pyrsistent?'), awkward co-installation, and a typosquatting-adjacent optics problem on PyPI. Recommendation: rename before first release (candidates in the spirit: 'objstore', 'graft', 'durable', 'keepsake') or at minimum secure the PyPI name immediately, put a disambiguation line at the top of the README, and never abbreviate to 'pyrs'.

## MVP roadmap (as originally drafted — see amendments)

v0.1 'the loop works' (6-8 weeks, pure Python, zero hard deps beyond msgspec + pyroaring + psutil): @pyr.entity slots-dataclass decorator; OID registry (WeakValueDictionary + __pyr_oid__ stamping); msgspec-msgpack entity records with '<qqq' headers; single-channel append-only files + binary transaction log + truncate-on-boot recovery + configurable fsync; JSON-lines type dictionary; store()/store_root()/restart-restore (the EclipseStore hello-world in 4 lines); explicit Lazy[T] + LazyReferenceManager (timeout + RSS quota); class-swap ghost/dirty-tracking SPIKE in week 1-2 (it is novel — if frozen/slots edge cases bite, v0.1 ships explicit Lazy + fingerprint-diff only); EntityMap with pyroaring bitmap indexes and the Condition AST (queries must be in the first release — shipping 'persistence without queries' repeats ZODB's arc); crash-consistency property tests (torn-tail fuzzing) from day one. v0.2 'search + columns' (next ~2 months): commit-record index-delta outbox + watermark protocol; SQLite FTS5 sidecar (pyr.FullText); Arrow ColumnarProjection with roaring tombstones, compaction, DuckDB registration and to_pandas/to_polars; legacy type mapping auto rules (add/remove/reorder/widen) + RenamedFrom markers; offline mark&sweep disk GC + file dissolving (GC.md as spec); single-file backup. v0.3: usearch vector indexes (pyr.Vector, numpy in/out); pyoxigraph triple projection + SPARQL + rdflib Store facade; refactorings-file escape hatch for class renames; gc.freeze integration; PersistentList/PersistentDict polish. v1.0 (~9-12 months): pyrsistance-graphql package (strawberry default_resolver + DataLoader + relay nodes) and pyrsistance-fastapi (router, lifespan, pydantic-settings); async facade hardening; cp313t/cp314t wheels (all chosen deps already support free-threading — launch feature, not retrofit); storage-format freeze + documented format spec; tantivy-py and sqlite-vec as extras behind SearchBackend/VectorBackend protocols. EXPLICIT PUNTS (with reasons): multi-channel parallel writes (GIL; format stays compatible), client-server/ZEO mode (Durus proves embedded-single-writer suffices; diskcache proves the market), MVCC time-travel/undo, incremental tri-color disk GC (offline is enough — its only job is reclaiming disk), cloud/SQL storage backends beyond the 3-method blob protocol seam (S3 + SQLite later; the other ~10 EclipseStore AFS targets are enterprise checklist items), JCache/Hibernate analog (no Python equivalent exists), REST storage inspector, wrapt-style transparent proxies (measured permanent 4x tax), free-threaded parallel query execution (measured 2.5x pointer-chasing penalty; columnar kernels already release the GIL).

## Open risks

Ranked: (1) Dirty-tracking completeness — the #1 DX killer in both ancestors (EclipseStore silently loses in-place child mutations; ZODB's _p_changed tax drove users to ORMs). Mitigation: tri-state class-swap as default, PersistentList/Dict as default container types, msgspec-fingerprint commit diff as the safety net for unswappable types, eager-storer escape hatch, and a debug mode that fingerprints everything and WARNS on detected-but-untracked mutations. (2) Class-swap ghost novelty — zero-overhead is verified on 3.14 benchmarks but unproven at library scale (frozen dataclasses, layout-incompatible __slots__, C-extension types, metaclass conflicts). Mitigation: it is an optimization, never load-bearing — explicit Lazy[T] (EclipseStore's own chosen design) is the documented API and full fallback; spike in v0.1 week 1 with a kill criterion. (3) Memory footprint 3.5-4x Java (10M objects ≈ 1.4-1.7 GB; pydantic would be 4x worse). Mitigation: slots+msgspec.Struct canonical form, Lazy boundaries on large collections (the GigaMap lesson), Arrow-backed columnar storage for value-only collections, RSS-quota clearing, and honest documentation of the budget — never market 'billions of objects in RAM'. (4) Homegrown crash consistency — storage engines earn trust slowly and one corruption report kills adoption. Mitigation: port EclipseStore's battle-tested recovery design verbatim (log-before-action, truncate-on-boot) rather than inventing; add the fsync EclipseStore lacks; property-based torn-write/kill -9 fuzzing in CI from v0.1; 'mirror/index = rebuildable derived data' so only the core log must be perfect. (5) Dependency abandonment — Kuzu's Oct 2025 archival is the cautionary tale; usearch is single-vendor, sqlite-vec single-maintainer, pyroaring small. Mitigation: every index is rebuildable from the graph behind a backend protocol (swap cost = rebuild, not migration); defaults biased to stdlib (sqlite3/FTS5) and multi-vendor Rust/PyO3 projects (tantivy, oxigraph); pin + vendor escape hatches. (6) Write throughput vs durability — per-commit fsync costs milliseconds where EclipseStore 'wins' benchmarks by not syncing at all. Mitigation: explicit policy triad (per-commit/interval/never) with group commit, documented honestly; the sidecar watermark protocol already tolerates lagging indexes. (7) Name collision with pyrsistent — reputational/discoverability risk, cheap to fix now, expensive after launch. Mitigation: rename or reserve PyPI name + disambiguation before any public artifact. (8) Adoption in a dormant niche — ZODB's 205K/month says 'object DB' alone won't sell. Mitigation: lead with the wedges nothing else has — FastAPI/GraphQL-native persistence with batched lazy resolution, and zero-copy DuckDB/pandas views of live objects — and position as 'SQLite for object graphs + search', not 'a better ZODB'.
