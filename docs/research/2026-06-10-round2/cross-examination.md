# Cross-examination of the round-2 recommendations

_A skeptical reviewer checked the five recommendations against each other and against the design constraints (solo maintainer, single-writer, local-first persona, ~24-month honest v1.0)._

## Verdicts

### rust-core — **adopt**

Internally sound and the only rec that takes the solo-maintainer constraint as the primary design force. Amdahl framing is correct: the data plane is already compiled (msgspec, CRoaring, DuckDB/Arrow, polars, usearch, SQLite), and the residual pure-Python work (dirty __setattr__, registry, swizzling) is PyObject-bound where PyO3 boundary cost erases gains. Dependency-health claims verified: redb's PyPI binding is frozen at v0.5.0 / Aug 2022 (https://pypi.org/project/redb/). Two hedges: (1) the 'by then PEP 803/abi3t will have collapsed the wheel matrix' claim is optimistic — PEP 803 exists but was still in 'round two' steering-council discussion as of the latest public record (https://peps.python.org/pep-0803/, https://discuss.python.org/t/pep-803-round-two-abi3t-stable-abi-for-free-threaded-builds/106181, https://github.com/python/cpython/issues/146636); do not schedule anything contingent on it. (2) Its 'projected 2-3.5s boot for 1M objects is acceptable' framing is superseded by startup-boot's persisted-index plan, which removes the strongest residual Rust motivation entirely — strengthening, not weakening, this rec's conclusion. Note 'pure-Python-first' really means 'numpy/Arrow-vectorized-first'; that is fine because Arrow mirrors are already decided core.

### distributed-serverless — **adopt-modified**

The layering analysis is correct and verified: Eclipse Data Grid (1 writer + N full-copy readers, event-stream replication) was indeed open-sourced July 2025 and took a funded company a decade (https://microstream.one/blog/2025/07/14/eclipse-data-grid-is-available-as-open-source/) — confirming 'everything beyond one writer took funded teams.' Three modifications: (1) MOVE UP the cheapest, highest-leverage item the rec only implies: an EclipseStore-style lease-refreshed lock file with a loud error message belongs in v0.x core (~1-2 days; port of StorageLockFileManager.java, verified at /Users/sh/pyrsistance/resources/store/storage/storage/src/main/java/org/eclipse/store/storage/types/StorageLockFileManager.java). 'uvicorn --workers 4' silently corrupting is the #1 foreseeable user error; documentation alone does not prevent it. (2) REPLACE v1.x 'S3 log shipping (nearly free on the append-only format)' — a sequencing error: the append-only format is a post-v1 optimization per the decided design, so v1.x shipping has nothing to ship. The v0.x SQLite-blob store gets PITR for free by documenting Litestream / sqlite3.backup (https://litestream.io) — a docs recipe, not a feature. (3) SCOPE DOWN v1.x read-only snapshot readers to 'open the store file read-only at a commit watermark, no live invalidation' — 'nearly free' underestimates per-process registry/swizzling plus a cross-process invalidation protocol, which is a mini replication system. DBOS command-queue recipe (https://github.com/dbos-inc/dbos-transact-py): adopt as documentation only.

### crdt — **adopt-modified**

The core stance — CRDTs solve cross-device merge, not same-machine multi-worker, and CRDT-as-core-data-model destroys the 600B envelope, unique indexes, and ref integrity — is correct and consistent with distributed-serverless. Ecosystem claims verified: cr-sqlite's last commit was 2024-10-25, i.e. ~20 months dormant as of 2026-06 (https://github.com/vlcn-io/cr-sqlite); pycrdt is the sole production-grade option (https://github.com/y-crdt/pycrdt). Modifications: (1) The Annotated[Text, pyr.Crdt] extension must NOT appear on any committed roadmap — within the ~24-month solo v1.0 envelope it is pure scope creep until demand exists; punt, design nothing for it beyond the watermark pipeline that already exists for FTS5/usearch. (2) Its 'optional writer-IPC command queue extension' duplicates distributed-serverless's user-land recipe — resolve in favor of recipe-only (one fewer owned component). (3) Flag for any future extension docs: a pycrdt doc blob per field carries 10-100x plaintext memory overhead, materially violating the 600B/object envelope for objects that use it — must be documented as outside the envelope.

### startup-boot — **adopt-modified**

The analysis is the strongest of the five: persisted indexes beat parallelism (measured 2-thread GIL scan 1.5x SLOWER), variable-length header chains cannot be vectorized so offsets must live out-of-band, and SQLite-blob v0.x has no boot problem at all. Adopt the never-list (thread-partitioned scans, multiprocessing channels) wholesale. Two modifications: (1) 'Freeze the custom-log record/trailer format now' contradicts rust-core's correct 'spend iteration speed on API design' ethos — freezing an on-disk format for an engine that does not exist is premature commitment; instead VERSION the format header and RESERVE the sealed-flag, footer-offset, and tid-watermark fields (near-zero cost, preserves the entire option). (2) 'Custom-log v1: implement BOTH levers' silently promotes the custom log from 'later optimization' (the decided design's words) into the v1 critical path — reject the promotion; the checkpoint+footer chain ships with the log whenever profiling justifies the log, which per rust-core is post-v1. Net effect: this rec's deliverable inside the 24-month window is ~3 reserved header fields and a benchmark harness.

### rdf-native — **adopt-modified**

The negative space is exactly right: no homegrown SPARQL/Cypher, no persistent RocksDB, triples as rebuildable sidecar; ecosystem claims verified — Kuzu archived 2025-10-10, Apple acquisition confirmed Feb 2026 via EU filing (https://betakit.com/apple-strikes-deal-to-acquire-canadian-database-software-startup-kuzu/, https://www.macobserver.com/news/apple-buys-graph-database-startup-kuzu-eu-filing-shows-more/). But 'CORE in v0.x' for the reverse-reference index is scope creep dressed as a feature, and the rec refutes itself twice: (a) its own rebuildable-derived-data argument means the index can be added in any later release with zero format risk — which is precisely the argument for NOT putting it in v0.x; (b) its strongest core justification, ghost-eviction safety, depends on class-swap ghosts which the decided design itself defers as 'optimization,' and cascade-delete/GC are not decided features at all. Move reverse index + a minimal traversal API (incoming() first) to v1 as a watermark-fed sidecar; +5-15% on the 600B envelope is honest and acceptable. The DuckPGQ recipe (https://duckpgq.org) is genuinely cheap (days, docs-only over already-decided Arrow mirrors) and moves UP into v1 docs. pyrsistance-rdf (~6-7 claimed weeks, realistically 2-3x for a solo maintainer): punt, demand-driven, no roadmap commitment.

## Contradictions found

(1) CUSTOM-LOG SEQUENCING — the sharpest conflict. Decided design: custom append-only log is a 'later optimization.' startup-boot promotes it to 'Custom-log v1: implement BOTH levers'; distributed-serverless's v1.x S3 log shipping presupposes it exists ('nearly free on the append-only format'); rust-core keeps it post-v1. Three recs, three timelines. Resolution: post-v1, per rust-core and the decided design; v0.x reserves format fields only; PITR meanwhile via Litestream over the SQLite-blob file (docs, not code). (2) COMMAND-QUEUE OWNERSHIP — distributed-serverless says user-land recipe (DBOS/Hatchet/Celery concurrency=1); crdt says 'optional writer-IPC command queue extension.' Same feature, two owners. Resolution: recipe-only; an owned extension is a second writer-coordination component for a solo maintainer to support forever. (3) RDF-NATIVE vs SCOPE REALISM AND ITSELF — 'reverse index in core v0.x' contradicts the solo/24-month constraint and the rec's own logic: rebuildable derived data is by definition addable later without migration risk, and the ghost-eviction-safety justification depends on ghosts the decided design defers. (4) BOOT-PROBLEM FRAMING — rust-core calls 2-3.5s/1M-object boot 'already acceptable' while startup-boot engineers two levers to make it 10-100ms; not a hard contradiction but resolves the same way: SQLite v0.x has no boot problem, so both reduce to 'do nothing now, reserve fields.' (5) AGGREGATE v1 SCOPE vs 24-MONTH ENVELOPE — the decided core (object engine + dirty tracking + Lazy + SQLite store + bitmap/Condition AST + Arrow mirrors + FTS5 + usearch + watermark pipeline + GraphQL + FastAPI) is already aggressive for one person; the recs collectively add reverse index + traversal API + custom log + footers + checkpoints + read-only readers + S3 shipping + edge-bitmap accelerators + two extension packages. Honoring all five 'v1/v1.x' lists blows the envelope ~2x; forced pruning below. (6) 'PURE-PYTHON-FIRST' vs DEPENDENCY WEIGHT — startup-boot's checkpoints (numpy/mmap) and rdf-native's reverse index (Arrow tables) make numpy/pyarrow load-bearing; consistent with the decided Arrow-mirror core but inconsistent with a slim local-first/CLI install unless query/search/vector/web ship as extras. (7) 'NEARLY FREE' CLAIMS — both distributed-serverless (read-only readers, S3 shipping) and rdf-native (traversal API '~1 week') systematically lowball integration cost; every 'nearly free' item rides the commit-delta/watermark pipeline, which all five recs treat as existing infrastructure — it is the single most load-bearing undelivered component in the design and a single point of rework if its semantics (idempotency, ordering, delta schema) are wrong. (8) MINOR FACT HEDGE — rust-core's reliance on PEP 803/abi3t landing in 3.15 is unconfirmed (still in steering-council discussion: https://github.com/python/steering-council/issues/333); harmless because nothing v1 depends on it.

## Revised priorities (ratified)

CORE v0.x (ordered): 1. Object engine: slots-dataclasses canon, msgspec msgpack records, WeakValueDictionary OID registry, tri-state dirty tracking, explicit Lazy[T] (ghosts stay deferred per decided design). 2. SQLite-as-blob-store behind the 3-method protocol + single-writer lease-refreshed lock file with loud error (NEW, moved up from implicit; ~1-2 days, port of EclipseStore StorageLockFileManager — /Users/sh/pyrsistance/resources/store/storage/storage/src/main/java/org/eclipse/store/storage/types/StorageLockFileManager.java). 3. Commit-delta/watermark pipeline specified, versioned, and tested as a PUBLIC contract (NEW emphasis: it is the substrate all five recs assume; idempotency + ordering semantics locked before any consumer ships). 4. pyroaring bitmap indexes + Condition AST (the differentiating query story). 5. Format hygiene: versioned (not frozen) custom-log record header with reserved sealed-flag/footer-offset/tid-watermark fields (~zero cost). 6. Docs: workers=1+asyncio deployment guide; DBOS/Celery concurrency=1 command-queue recipe; Litestream + sqlite3.backup PITR recipe (replaces all v1.x S3-shipping code). CORE v1: 7. Arrow columnar mirrors + DuckDB/polars zero-copy queries (decided design). 8. Reverse-reference index as rebuildable Arrow sidecar on the watermark pipeline + minimal traversal API (incoming() first) — moved DOWN from rdf-native's v0.x, justified by backlinks/orphan checks once the engine is stable. 9. DuckPGQ property-graph recipe over the Arrow mirrors (docs-only, days — moved UP from extension tier). EXTENSION PACKAGES (separate wheels/extras, decided features but not in the slim core install, sequenced after v1 core freeze): 10. pyrsistance[fts] — SQLite FTS5 sidecar (cheapest: stdlib). 11. pyrsistance[vector] — usearch sidecar. 12. pyrsistance[web] — FastAPI + strawberry GraphQL integration. 13. v1.x: read-only snapshot readers, scoped to open-at-watermark with no live invalidation. PUNTED — demand-driven, zero roadmap commitment: 14. Custom append-only log + footer/checkpoint boot chain (only on profiling evidence from real workloads). 15. Optional Rust 'turbo' wheel for log scan/compaction behind the same 3-method protocol (only after #14, and only if PEP 803-class ABI relief materializes). 16. S3 log shipping / SlateDB-style lease-fencing extension (requires #14). 17. pyrsistance-rdf (term dictionary, triples sidecar, rdflib facade, optional in-memory pyoxigraph). 18. CRDT field extension (pycrdt blob fields). NEVER (all five recs agree, ratified): Rust for dirty-tracking/registry/swizzling/Condition AST; redb/fjall/sled/RocksDB storage; CRDT as core data model; clustering/FaaS scale-out/multi-writer in core; GIL-thread-partitioned boot scans; homegrown SPARQL or Cypher engine.
