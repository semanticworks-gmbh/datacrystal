# Adversarial critique — merged plan

The merged plan is largely faithful to the ratified docs — M0 contains all the essentials (PyPI reservation, fitness harness, header-v1 reserved fields), no Never-list or punted item is smuggled in, and the tracer-spine sequencing mostly respects ADR-001 and the SDA deltas. But it breaks in five real places: ruling 3 justifies a permanent eager commit path by misreading DESIGN.md's "eager storer opt-in" (which is a traversal-depth variant of the buffer-until-commit storer, not an eager-I/O commit); M3's snapshot deliverable depends on M4's bitmap indexes (dependency inversion); the COMMIT-DELTA lock at M3 precedes the one consumer (bitmap index) whose update/delete un-indexing semantics are most likely to expose schema gaps — recreating the contract-frozen-wrong failure mode the plan claims to kill; the #1-risk mitigations (PersistentList/Dict, fingerprint diff, debug=True) are named in the risk register but scheduled in no milestone; and 26.5 dev-days contradicts the ratified "v0.1 ≈ 2–3 months" timeline-honesty amendment for a scope that has grown since that estimate.

## [major] Ruling 3 misreads the ratified 'eager storer'. DESIGN.md defines it inside the buffer-until-commit storer's semantics: 'buffer-until-commit storer with EclipseStore semantics (argument always rewritten, registered children skipped; eager storer opt-in)' — i.e., an eager-traversal variant (re-store registered children), not an eager-I/O commit path. Declaring M1's single-phase eager commit() 'the ratified opt-in eager storer, kept forever' creates a permanent second commit path that bypasses three-phase commit, which ADR-001 never sanctioned ('The buffer-until-commit storer is built in this shape from its first version') and which would freeze wrong public commit semantics into v0.1.

**Fix:** Relabel M1's eager commit() as an explicitly temporary tracer scaffold that is deleted/absorbed when the three-phase storer lands at M2 (or build M1's commit trivially three-phase-shaped from the start). If an opt-in eager storer ships, implement it as the eager-traversal option within the three-phase buffer-until-commit storer, matching DESIGN.md's definition.

## [major] Dependency inversion: M3 promises 'store.snapshot() (frozen-DTO + frozen-bitmap views at watermarks)' but pyroaring bitmap indexes are built in M4. ADR-001 bound decision 4 and ROADMAP item 3 define the snapshot views as 'frozen-DTO record reads + frozen roaring bitmaps at commit watermarks' — the bitmaps are the index bitmaps, which do not exist at M3. M3's exit criterion ('snapshot reads from a thread pool during owner commits') silently drops the bitmap half, so either M3 claims an undeliverable artifact or v0.1 ships snapshot bitmaps untested.

**Fix:** Rescope M3 snapshot to frozen-DTO views + watermark with the bitmap-view API slot explicitly reserved (or per-type extent bitmaps only, stated as such); add to M4's exit criteria 'snapshot exposes frozen index-bitmap views; a foreign thread runs a bitmap query against a snapshot during an owner commit'. ADR rider 2 is still satisfied at the v0.1.0 tag.

## [major] The COMMIT-DELTA-v1 lock at M3 exit happens before the consumer most likely to break the schema validates it. Snapshot and the toy counter are read/accumulate-shaped; a bitmap index (M4) is the first consumer needing un-indexing on update/delete — i.e., the delta must carry prior values (or equivalent per-field old/new or tombstone semantics) or the schema is wrong. Discovering that in M4, after byte-pinned version-bump-or-fail CI, forces COMMIT-DELTA-v2 inside v0.1 — exactly the 'contract-frozen-wrong' failure mode ruling 1 claims to kill. ROADMAP item 3's 'locked before any consumer ships' does not require locking before in-tree consumers are *built* against the draft.

**Fix:** Move the lock to the v0.1.0 tag (M4 exit), after the bitmap consumer has run against the draft — both in-tree consumers then ship simultaneously with the lock, still strictly before the external FTS consumer (late v0.x). Alternatively keep the M3 lock but add a mandatory index-shaped consumer spike (update/delete un-indexing over the draft deltas) to M3 pre-lock, plus an evil twin that fails when prior-value information is missing.

## [major] The #1 risk's mitigations are listed but never scheduled. PersistentList/PersistentDict appear only in the repo-layout comment and risk register; the msgspec-fingerprint-diff safety net and ADR-001's debug=True mode ('debug=True keeps hooks permanently armed') appear in no milestone deliverable, day plan, or exit criterion. DESIGN.md mandates 'ship PersistentList/PersistentDict as default container field types' and names in-place child mutation loss the #1 DX killer; D7's single day covers only the setattr hook + state machine, so container mutations — the exact EclipseStore footgun — are untracked and untested in v0.1 as planned.

**Fix:** Add PersistentList/PersistentDict, the fingerprint-diff commit safety net, and debug=True (permanently-armed hooks + warn-on-untracked-mutation) as named M2 deliverables; extend the RuleBasedStateMachine with list/dict child-mutation operations vs the dict oracle as an M2 exit criterion; budget +2–3 days.

## [major] Effort contradicts the ratified timeline-honesty amendment. DESIGN.md amendment 9: 'v0.1 ≈ 2–3 months' — for a smaller scope than this plan (which has since absorbed confinement guards, three-phase commit, the public contract + conformance kit + evil twins, snapshots, unique index, frozen mode, lease lock, journal, executable docs). The plan claims 26.5 dev-days (~5.5 weeks). M1 packs entity decorator, registry, codec+swizzling+checksums, type dictionary, storage protocol + two backends, store facade, hydration, Lazy[T], owner binding, and the SIGKILL demo into 5 days (D6 alone holds registry, lazy, store, demo, and the integration test); M2 packs 11 deliverables including drafting 'the single most load-bearing undelivered component' into 7.5 days.

**Fix:** Re-baseline to ~45–55 dev-days (or explicitly trim: e.g., evil twins and async demo polish slide into M4), keep '2–3 months' as the external commitment, and split D6 into two days with the M1 tag at D7.

## [minor] M2 exit criterion 'journal starts passing' is untestable as written: the journal's defined scenes include unique slugs (M4), tag queries (M4), and snapshot() report from a worker thread (M3), none of which exist at M2. 'Starts passing' specifies no measurable subset.

**Fix:** Enumerate the journal scenes gating M2 (frozen Event log mutation-raises, Lazy[T] attachments, get_many, run-twice persistence) and mark the M3/M4 scenes as xfail-with-milestone-tag so the M2 gate is a concrete green/red signal.

## [minor] DESIGN.md amendment 7's 'defined behavior for old-code-opens-newer-store' has no v0.1 deliverable or exit criterion. M0 lays down header-v1 constants and M1 versioned records, but no milestone tests the refusal path; without it, the first post-v0.1 format bump becomes an undefined-behavior compatibility break against stores in the wild.

**Fix:** Add to M1's exit criteria: opening a store whose header/record version exceeds the library's supported version raises a specific loud error (e.g., NewerStoreError naming both versions); pin a fixture store with a bumped version byte in the conformance suite.

## [minor] LazyReferenceManager (M2) has no exit criterion covering ADR-001 race (c) and bound decision 3 ('NEVER touches the live graph from a foreign thread — loop task in async mode, sentinel-posts + owner-side piggyback in sync mode'), and its DESIGN-spec RSS-quota clearing requires psutil, which the plan's own fitness gate (deps exactly {msgspec, pyroaring}) forbids — an unresolved contradiction left implicit.

**Fix:** Add an M2 exit test that demotion only ever executes on the owner (sentinel-post path exercised under load while the owner reads), and state explicitly that v0.1's manager is timeout-only with RSS-quota clearing deferred (or stdlib-based) — recording the psutil exclusion as a decision, not an accident.

## [minor] The journal's 'get_many backlinks' wording flirts with ROADMAP item 8 — the reverse-reference index ('buys backlinks') is core v1, not v0.x. If the example implies engine-derived incoming references, it is scope creep; if app-maintained OID lists hydrated via get_many, it is legal but the wording invites the wrong reading and a premature API expectation.

**Fix:** Reword to 'get_many over app-maintained backlink OID lists' in the plan and journal docstring, and add a one-line note that engine-derived incoming() traversal is v1 (ROADMAP item 8) so the example is never mistaken for a reverse-reference feature.
