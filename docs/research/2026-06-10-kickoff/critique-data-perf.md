# Adversarial critique — dataset + perf suite

The dataset recommendation survives the license/data attack: I independently re-verified Wikidata's licensing page ("All structured data in the main, property and lexeme namespaces is made available under the Creative Commons CC0 License"), reproduced the headline count live on WDQS (exactly 6,285 items with P31=Q12089225 "mineral species"), confirmed P2695 = "type locality (geology)", and confirmed the Wikipedia-extracts (CC BY-SA 4.0) and MovieLens ("may not use ... for any commercial or revenue-bearing purposes without first obtaining permission") characterizations — and the graph is genuinely not flat (Lazy single refs, list-of-refs, unique keys, frozen entity all real). The damage is elsewhere: (A) and (B) contradict each other about what domain the benchmarks actually run on, (A)'s flagship query is a cross-entity join the ratified v0.x Condition AST cannot execute, and several perf gates measure warm page cache labeled "cold" or compare against floors (1-object encode, pyroaring AND at 100k, two-sided fsync bounds) that are miscalibrated or will flake on shared runners.

## [major] (A) and (B) define two different benchmark domains and both claim to be THE generator. (A) 'Scaling into the perf generator' says the vendored mineral data IS the generator vocabulary (1M preset = 7.8k backbone + 450k Specimen + ~540k CatalogEvent, queries on Specimen.quality/mass_g) and that 'README demo, integration tests, FTS harness, and benchmarks share one language'; (B) §2 defines benchmarks/_domain.py as 70% Person / 20% frozen Note / 8% Tag / 2% City with 'Berlin' predicates, slugs, manager chains depth ≤6, and 0.1% cycles. Entity mix, facet cardinalities, ref topology, frozen ratio, and the canonical gate predicates all differ — the suite's corpus identity is undefined, and the deep-traversal/cycle topology (B) needs exists in neither vendored data nor (A)'s Specimen layer.

**Fix:** Pick one domain before any code: port (B)'s topology requirements into the mineral-cabinet generator (manager-chain analog: Specimen provenance/exchange chains depth ≤6, ~0.1% reference cycles, 10 'mega' minerals as hub fan-in, a ~1%-selectivity engineered predicate on Specimen fields), keep Person/Berlin only as DESIGN.md prose; or explicitly declare the perf domain separate and delete (A)'s 'benchmarks share one language' and 1M-preset paragraph.

## [major] (A)'s flagship query `(Specimen.quality == 'A') & (Mineral.crystal_system == 'cubic')` is a cross-entity join. The ratified v0.x query story (DESIGN.md tier 1, ROADMAP item 4) is per-entity-class bitmap indexes + Condition AST returning a bitmap of ids of ONE class; ANDing a Specimen-space bitmap with a Mineral-space bitmap is meaningless, and joins are explicitly tier-2 (Arrow/DuckDB, v1) / DuckPGQ (v1, docs-only). The README/demo centerpiece query cannot run on the first runnable version, and (A) claims it 'runs unchanged at every scale'.

**Fix:** Replace with a single-class predicate (e.g. `(Specimen.quality == 'A') & (Specimen.mass_g >= 100.0)`), or denormalize crystal_system onto Specimen in the demo model; if a semi-join is wanted, spec it honestly as a ref-OID-bitmap predicate (`Specimen.mineral.in_(bitmap_from_mineral_query)`) and get it ratified as a Condition AST extension first.

## [major] Boot benchmarks measure warm OS page cache while labeled 'cold open→root-ready'. Store files are 'built once per session and cached in tmp', the bench runs in a fresh subprocess (which clears the process, not the page cache), and (B) takes the *min* of 5 repeats — systematically selecting the most-cached run. This is exactly 'cached SQLite page reads pretending to be cold boot': the O(checkpoint) envelope is only ever validated in the CPU-bound cached regime, where superlinear cold-I/O effects (B-tree depth, readahead patterns) are invisible.

**Fix:** Rename the metric to warm-cache boot and document it (the ratio gates boot(10N)/boot(N) and boot_vs_history remain mostly valid within that regime), and add a nightly true-cold variant: copy the store file to a fresh inode then evict (Linux CI: `posix_fadvise(POSIX_FADV_DONTNEED)` on the copy or cgroup memory pressure; macOS dev: `purge`), gated as its own same-run ratio pair.

## [major] commit_latency_fsync gates will flake or are miscalibrated by construction, on every PR. (1) Two-sided bound: t(per-commit) − t(never) within 0.5–2× raw os.fsync(4 KB) — a difference of two noisy quantities against a floor that is itself bimodal on shared runners (virtio/overlayfs fsync ranges ~100 µs to ~10 ms run-to-run; macOS F_FULLFSYNC is another regime); the 0.5× lower bound fails legitimately if SQLite coalesces journal syncs. (2) t(never) ≤ 5× 'encode floor': a 1-object msgspec encode is ~sub-µs while any real 1-object commit carries fixed SQLite BEGIN/COMMIT + lease/watermark bookkeeping in the tens of µs — permanently red regardless of engine quality. Related: commit_tput_small's floor ('executemany of identical payloads') does not pin transaction-boundary parity (100 separate floor txns vs one batch) or fsync policy parity with the engine run.

**Fix:** Gate only the policy ordering (t(per-commit) > t(interval) ≥ t(never)) plus a one-sided sanity check (difference ≥ ~0.3× same-run fsync floor) and move it to the nightly pinned runner; define t(never)'s floor as encode + a real 1-row SQLite transaction (BEGIN/INSERT/COMMIT, synchronous=OFF); write floor-parity rules into the suite spec: every floor replicates the engine run's transaction boundaries and fsync policy exactly.

## [major] query_bitmap_vs_scan's 'query ≤ 20× same-run bare pyroaring AND' transplants a constant calibrated at the 306 µs/10M-id anchor to PR scale (100k), where the bare AND floor shrinks to ~1–5 µs but the engine's fixed per-query Python overhead (AST walk, index lookups, result wrapping — tens of µs) does not shrink with N. The ratio at 100k lands anywhere between ~10× and ~100× depending on constant overhead, so the gate fails or passes on fixed cost, not regression — likely permanently red at PR scale or tuned into meaninglessness by the 2×-noise headroom rule.

**Fix:** Keep the floor-ratio gate only at 1M nightly; at PR scale gate the absolute overhead (t_query − t_bare_AND, same run) against a fixed per-query budget (e.g. ≤ 100 µs) or track it as trend-only, alongside the already-sound scan-speedup ratio gates (≥10×/≥50×).

## [minor] watermark_apply_fixed_delta's t(big)/t(small) ≤ 1.2 will spike non-deterministically once the FTS5 sidecar rides the delta ('once it lands'): FTS5 automerge triggers on cumulative segment history, so one 1k-op delta at the 1M corpus can absorb a multi-level segment merge that the 10k corpus never sees — blowing 1.2 on a randomly-chosen run of the suite's single most load-bearing O() proof, which also means the 14-green-nights warning-to-gate promotion will stall on it.

**Fix:** Gate the bitmap-index-only delta path at 1.2; for the FTS5-inclusive path, pin merge behavior (set 'automerge'/'usermerge' explicitly, or run a 'merge' to quiescence before timing) and gate the median over ≥10 consecutive deltas instead of one.

## [minor] The gating mechanism as written — 'two benchmark.pedantic-style timed blocks compared directly' inside one test — is not implementable with pytest-benchmark: the benchmark fixture raises FixtureAlreadyUsed on second use per test, so every same-run engine/floor ratio needs a hand-rolled timer for the floor side anyway, quietly contradicting the 'pytest-benchmark, not a custom timer' justification and leaving floor timing without the calibration/GC-off/outlier treatment the engine side gets.

**Fix:** Spec it explicitly: engine side rides the benchmark fixture (trend JSON); floors are timed by one small shared helper (perf_counter loop with gc disabled, same rounds/warmup constants as the pedantic config) provided as a session fixture, and the ratio assert lives in the test body.

## [minor] 'Same (seed, N, generator_version) ⇒ byte-identical store' is an unproven overclaim with hidden non-determinism sources unaddressed: (a) DESIGN.md ports EclipseStore's partitioned TID space, and EclipseStore TIDs are wall-clock-derived — if datacrystal TIDs carry timestamps, store bytes differ every run; (b) PYTHONHASHSEED is never pinned anywhere in (B), so any set iteration in generation or commit batching breaks byte-identity; (c) SQLite page layout varies across library versions and the cache key (seed, N, generator_version, format version) omits sqlite/python versions.

**Fix:** Downgrade the claim to 'semantically identical' unless TIDs are spec'd sequence-derived; pin PYTHONHASHSEED=0 in the bench environment and add a no-set-iteration rule to the generator; extend the fixture cache key with sqlite3.sqlite_version and the CPython version, and verify with a content hash recorded at first build.

## [minor] RSS measurement via resource.getrusage cannot implement the stated metric: ru_maxrss is the monotonic high-water mark, so '(RSS_after − RSS_before)/N' either doesn't work at all (before/after deltas of a monotonic counter) or silently reports the build-phase peak (transient encode buffers, generator scratch) instead of steady-state live-graph residency, inflating bytes/object against the 690 B gate; ru_maxrss is also bytes on macOS vs kilobytes on Linux.

**Fix:** Read current RSS (psutil.Process().memory_info().rss — psutil is already in the bench group) before and after corpus registration, after gc.collect() and an allocator trim; keep ru_maxrss only as a supplementary peak trend.

## [minor] The acquisition recipe pages WDQS with 'LIMIT 2000 OFFSET n' and no ORDER BY: SPARQL result order is unspecified without ORDER BY, so pages can overlap or skip rows between requests (especially with the label service and OPTIONALs in the pattern), making regenerate.py able to silently drop or duplicate minerals — undermining the 're-runnable' snapshot claim.

**Fix:** Add ORDER BY ?m (and deterministic ordering keys) to all three .sparql files, page on the ordered QID, and have regenerate.py assert the final distinct count against the expected total (currently 6,285) before writing the msgpack files.

## [minor] The CC BY-SA extracts handling is thinner than the verified attribution requirements: Wikipedia reuse requires per-page credit (hyperlink/URL to each source article), an indicate-changes statement (intro extracts are excerpts, i.e. modifications), and a license notice with each distributed copy. A bare 'ATTRIBUTION.md (CC BY-SA 4.0)' doesn't say this, and nothing specifies whether data/minerals/ — including extracts.msgpack — ships in the sdist/wheel, which would put share-alike content inside a package whose metadata will presumably declare a single permissive license.

**Fix:** Spec ATTRIBUTION.md to list per-mineral source-article URLs, the CC BY-SA 4.0 link, and an 'excerpted intro paragraphs, otherwise unmodified' changes note; exclude extracts.msgpack from the wheel/sdist (fetch-on-demand via regenerate.py) or explicitly declare the mixed licensing in package metadata before the PyPI name goes live.

## [minor] (B) violates its own headline principle ('never absolute wall-clock') in §2: 'generation time itself is asserted ≤ 30 s @1M as a tripwire' is an absolute wall-clock gate that will flake on contended shared runners (a 3× slowdown over the ~10 s expectation is routine on busy CI), failing nightly runs for reasons unrelated to any regression.

**Fix:** Make the tripwire a same-run ratio against a trivial floor measured in the same process (e.g. generation time ≤ K× the time to msgspec-encode the same object count), or demote it to a trend alert via extra_info like the other absolute numbers.
