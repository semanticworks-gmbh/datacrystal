# Adversarial critique — fitness functions

The catalog is fundamentally sound — the M0-five sequencing is correct, the gate/trend split mostly follows its own ratio-not-absolute principle, and the Never list and dep boundaries are well covered. But under adversarial reading, several perf gates are mis-instrumented as written (timing nanosecond/microsecond operations against 5%-2x thresholds, tracemalloc contaminating the RSS it measures, an absolute 100ms wall-clock PR gate that violates the doc's own anti-flakiness rule), the ADR-001 suite claims all six bound sub-decisions while silently omitting the daemon principle (race c), and the versioning promises are tested in one direction only — old-code-opens-newer-store (DESIGN amendment 7) and sidecar-ahead-of-core detection (amendment 6) have no fitness function at all.

## [major] #4 ADR-001 conformance claims to cover 'all six bound sub-decisions' but the mechanism lists no check for sub-decision 3, the daemon principle — 'the LazyReferenceManager NEVER touches the live graph from a foreign thread'. Race (c) (use-after-clear during demotion) is one of the five races ADR-001 exists to close and explicitly 'bites even single-threaded apps', yet nothing in the suite exercises demotion at all. The gate goes green while the single most subtle contract clause erodes.

**Fix:** Add to the conformance suite: instrument the ghost-demotion path to record the acting thread/task id and assert it is always the owner (or the owner-side piggyback) in both sync and async modes; plus a stress case where the manager's timeout/RSS trigger fires while the owner reads the same entity under debug=True, asserting no use-after-clear.

## [major] #12 import-time budget gates every PR on '≤ 100 ms CI runner' — an absolute wall-clock threshold on shared GitHub runners, the exact flaky-by-construction pattern Section 3 itself bans for nightly perf ('avoids flaky absolute thresholds on shared runners'). The '≤ 1.3× stored baseline' half compares against a measurement from a different run/machine, which is the same cross-machine flakiness with extra steps.

**Fix:** Make the PR gate a same-run ratio: measure cold-subprocess import of datacrystal vs a fixed reference import (e.g. `import msgspec` or bare interpreter startup) in the same job and gate on that ratio; keep the 100 ms number as a nightly trend alert on a pinned runner class, and archive -X importtime profiles for the trend as already specified.

## [major] #15 steady-state overhead gates at ≤ 1.05× plain dataclass for attribute reads/writes — a 5% threshold on ~20-50 ns operations. Even within one pytest-benchmark run, ns-scale comparisons swing more than 5% from CPU frequency scaling, core migration, and cache state on shared runners; this gate will cry wolf and the threshold will get quietly widened, after which it guards nothing. The actual ratified property ('zero steady-state cost, one-shot __setattr__') is structural, not statistical.

**Fix:** Replace the primary gate with deterministic structural assertions: after the one-shot hook fires/disarms, assert `type(entity).__setattr__` is the original slot descriptor path and Clean-entity `__getattribute__` is unhooked (identity comparison against the plain class's attributes); keep the pytest-benchmark numbers as nightly Trend with a looser (e.g. 1.25×) alert, not a Gate.

## [major] #11 memory envelope measures 'RSS delta + tracemalloc' in the same run. Enabling tracemalloc allocates trace bookkeeping for every allocation (tens of bytes per object at 1M objects), so the RSS delta it reports includes tracemalloc's own overhead — the 600 B/obj gate is contaminated by its measurement instrument and will either fail spuriously or get widened to absorb the instrument, silently breaking the 'envelope never widened silently' promise. Allocator/platform variance (ubuntu glibc vs macOS) also makes one absolute byte threshold wrong on at least one OS.

**Fix:** Run two separate child processes: one plain run measuring RSS delta only (the gating number, with per-platform baselines), one tracemalloc run used only for attribution/bisection when the gate fails. Gate per-platform against checked-in baselines with a ratio (e.g. ≤ 1.1× baseline) plus the absolute 600 B as the documented ceiling on the reference platform.

## [major] #9 O(delta) gate times a fixed 100-record apply — microseconds for bitmap deltas, low milliseconds for FTS5 — and gates on the ratio of two tiny noisy wall-clock numbers (t(10⁶)/t(10³) ≤ 2×). Ratio framing does not rescue measurements where scheduler jitter exceeds the signal; this is flaky-by-construction at the stated threshold. Separately, rebuilding 10⁶-entity stores from scratch every night is the expensive part of the check while contributing nothing to the property under test.

**Fix:** Gate on operation counts, not time: the catalog already builds a counting storage wrapper for #14 — assert the number of storage reads/writes/rows touched during apply is a function of delta size only (identical across 10³/10⁵/10⁶ stores, or bounded by delta·log n with explicit allowance). Keep wall-clock as Trend only. Pre-build the three reference stores once and cache them as versioned CI artifacts (regenerate on format-version bump).

## [major] #1 crash-torture's regression mechanism is partially unreproducible as described: 'real SIGKILL at randomized commit-phase boundaries' depends on OS scheduling, so 'failing seed committed as regression case' cannot replay the SIGKILL timing from a hypothesis seed. Also 'assert content == last acked watermark, byte-exact' is undefined for the SQLite-as-blob backend — post-recovery file bytes legitimately differ (WAL/journal/freelist state) from any canonical form, so byte-comparing the store is wrong and comparing 'content' is unspecified. Third, 200 examples each spawning a child process + store open/close threatens the < 3 min full-gate budget on macOS runners by itself.

**Fix:** Make the fault plan explicit hypothesis-generated data (op index, commit phase, byte offset, fault type) that the child self-triggers via the FaultyStorage hook and os._exit — fully seed-reproducible and shrinkable; keep real SIGKILL as a nightly smoke layer without seed claims. Define the assertion as logical-state equality: decode all entities after reopen and compare against a shadow model at the last acked watermark (byte-exact on the re-encoded msgspec records, never on the SQLite file). Time-box the PR profile (hypothesis deadline / batch several examples per child process) and verify the 3-min budget in CI.

## [major] Coverage gap — versioning is tested in one direction only. #6 goldens prove new code opens old stores, but DESIGN amendment 7 ratifies 'defined behavior for old-code-opens-newer-store' and nothing checks it; similarly the commit-delta pipeline is a 'public versioned contract' (ROADMAP item 3) but no check asserts that a consumer encountering a newer pipeline version fails cleanly rather than misreading deltas. Both are exactly the silent-format-break failure modes the catalog claims to guard.

**Fix:** Add to release.yml: in a clean venv, `pip install datacrystal==<previous release>` and open the NEW release's golden store, asserting the documented behavior (loud versioned error or defined read-only mode per amendment 7); and feed a delta stream with an incremented pipeline-version field to the previous reference consumer, asserting clean rejection. Cheap PR-cadence variant: current code opening a fixture with version byte = current+1.

## [major] Coverage gap — DESIGN amendment 6's 'sidecar-ahead-of-core detection forces index rebuild' has no fitness function. #13 tests delete-sidecar-and-rebuild (sidecar behind/missing), but the dangerous direction is the opposite: core restored from backup (the documented Litestream/sqlite3.backup PITR recipe in ROADMAP item 6 makes this a first-class workflow) while the sidecar watermark is ahead — an undetected stale-ahead sidecar serves wrong query results indefinitely.

**Fix:** Add a scenario to #13: snapshot the store dir, advance N commits (sidecar applies them), restore the older core files under the newer sidecar, reopen — assert the watermark regression is detected, a rebuild is forced, and post-rebuild query results equal a from-scratch rebuild.

## [minor] #7 format-header lock is gameable exactly where it matters: the test only compares current encoding against the checked-in fixture, so a PR that changes the header layout AND regenerates the fixture passes green with the version byte unchanged. The 'change requires version increment' rule is process prose, not mechanism — the gate cannot see that the fixture moved without a version bump.

**Fix:** Check in an append-only registry mapping format-version byte → fixture content hash; the test asserts (a) current encoding matches the fixture, (b) the fixture hash for every previously-registered version is unchanged, (c) a new/changed fixture requires a new version entry. Changing bytes under an existing version then fails mechanically regardless of fixture regeneration.

## [minor] #2's 'format fixtures scanned for pickle protocol magic bytes' will false-positive on legitimate msgpack: pickle's magic is just b'\x80' + a protocol byte, and 0x80 is msgpack's fixmap(0) (any empty dict field), with 0x02-0x05 being common positive fixints right after. A two-byte scan over arbitrary msgpack streams is noise, and the resulting spurious red gates will train people to ignore or weaken the check.

**Fix:** Restrict the magic-byte scan to record-body offsets where a pickle payload could actually start (record start / blob field start), or replace it with a positive check: every fixture record must decode cleanly under the strict msgspec schema (validating decode is already 'structurally incapable of code execution' — decodability under the whitelist is the stronger property). Keep the AST import ban as the primary mechanism.

## [minor] #8 has a measurability problem and a missing scenario. 'Kill holder + age lease' requires either a real wall-clock wait of the lease TTL inside the PR gate or an unstated injectable clock. And the nastier failure mode is untested: a paused holder (GC pause, laptop sleep — core persona is desktop) whose lease expires and is taken over, then resumes and keeps writing — silent two-writer corruption, the exact thing the lease exists to prevent.

**Fix:** Make lease TTL and clock injectable in the test config (sub-second TTL for the PR gate, no sleeps near real TTL). Add scenario 4: suspend the holder (SIGSTOP), let a second process take over after expiry, SIGCONT the original — assert its next refresh/write raises a loud lease-lost error (the EclipseStore StorageLockFileManager self-check being ported) and the store is uncorrupted.

## [minor] #10 boot O(checkpoint) is near-vacuous for its entire stated lifetime: ROADMAP punt 14 says 'SQLite-blob v0.x has no boot problem — boot index *is* the B-tree', and with newest-wins one-row-per-OID storage the 1×/10×/100× churn stores are structurally identical on disk, so the ratio gate tests SQLite's B-tree, not datacrystal. Meanwhile it burns nightly budget generating ~10⁶ update operations, and the component it actually guards (custom append-log boot chain) is punted post-v1.

**Fix:** Replace the timing mechanism with a counting-wrapper assertion that does guard something now: open() must issue O(live-set) storage reads independent of total historical write count and dead-version rows (catches anyone accidentally introducing a full-history scan or version-keeping rows). Park the churn-ratio timing variant in the harness, activated as a gate only when ROADMAP item 14 (custom log) is picked up; until then drop it from nightly.
