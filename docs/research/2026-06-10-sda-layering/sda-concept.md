# SDA reading: sda-concept

_SDA layering study, 2026-06-10. Sources: /Users/sh/tetrahedron (concept docs, sda_store implementation, specs)._

## Summary

SDA ("Semantic Digital Asset", Hodapp 2014 thesis / 2018 workshop paper) is a semiotic data model: every piece of information is an immutable atomic record, a 7-tuple (sigmatics, pragmatism, semantics, syntax, asset, fingerprint, editstamp) — identity, functional role, meaning, encoding, serialized payload, SHA-256 hash, creation timestamp. A store is an append-only event log with exactly two operations (emit/revoke); complex documents are flat collections of SDAs sharing a sigmatics (multi-faceted identity), recursively decomposable (document → paragraph → sentence → NER annotation), and queryable orthogonally on any of the four semiotic axes. On top of the core tuple sit conventional layers expressed as pragmatism values: identifiers (uid dedup keys), owl/sameAs (instance equivalence), skos/exactMatch (concept equivalence), dcterms/subject (classification), plus an outline hierarchy (has_parents/has_children) and SKOS ontology concepts stored as SDAs themselves. Two application thrusts drive storage requirements: (a) GoBD-compliant archiving (German tax-audit immutability/completeness/traceability/machine-readability, extended with a Merkle-tree audit layer, retention timers, legal hold, Z3 CSV export) and (b) "semiotic retrieval" — a three-layer search architecture (FTS5+SKOS synonyms, LLM reasoning agent over axes+outline, dual per-record embeddings "Triple Sigmatics"). Conceptually SDA is positioned against RDF triples, Datomic datoms, event sourcing, standoff annotation, and content-addressable storage; an RDF projection (sda: OWL vocabulary, hybrid "Approach C": SDA store primary, RDF as query/linking layer) is planned, not core. In datacrystal terms, the SDA tuple is a single fixed-schema record type plus categorical indexes — most of its query needs map directly onto datacrystal's planned primitives, but its storage contract demands append-only immutability, content hashing at commit, and a verifiable completeness/audit chain that datacrystal does not currently promise.

## Key findings

### The SDA is a fixed 7-tuple, not a graph or schema framework

CONCEPT.md:13-25 defines SDA = (sigmatics, pragmatism, semantics, syntax, asset, fingerprint, editstamp). Sigmatics = URI-compatible identity ('what entity?'); pragmatism = functional role ('sentence', 'annotation', 'topology', 'provenance', 'identifiers', 'owl', 'skos', 'dcterms'); semantics = meaning within that role ('original', 'translation', 'entity-reference', 'sameAs', 'subject'); syntax = encoding (MIME types, language codes like 'en'/'de', DSL ids like 'topology:item'); asset = serialized payload (text/DSL/blob); fingerprint = SHA-256 of asset computed at emit; editstamp = temporal ordering. All dimension values are open strings — 'schema-free' is listed as an explicit limitation (CONCEPT.md:368).

### Invariants and lifecycle: append-only event sourcing with only emit and revoke

Core principles (CONCEPT.md:153-160): flat over nested, immutability (SDAs never modified — changes create new assets), event sourcing (emit = add, revoke = remove-marker that is itself a new record), orthogonal querying, multi-faceted elements, recursive decomposition. Revoke grows the log, never shrinks it (gobd-ledger-research.md:207). 'Current state' is a projection (DocElem grouping by sigmatics, CONCEPT.md:277-281). The only true deletion is retention-expiry deletion with documented Merkle-tree rebuild (gobd-ledger-research.md:224-228). Explicitly modeled on Datomic datoms (entity≈sigmatics, attribute≈pragmatism+semantics, transaction≈editstamp, added?≈emit/revoke; CONCEPT.md:252-270).

### Multi-faceted identity is the central design move

One sigmatics, many SDAs: a ZUGFeRD invoice is archived as 4 SDAs sharing sigmatics 'inv-2026-042' — original CII-XML, PDF/A visual, JSON classification annotation, JSON booking entry (gobd-ledger-research.md:253-258). Recursive decomposition example CONCEPT.md:135-145: HTML doc → paragraphs → sentences → FOODON entity-reference, all flat records linked by hierarchical sigmatics or topology SDAs. Multiple annotations on the same span are disambiguated by fingerprint (CONCEPT.md:307-314). This is standoff annotation generalized to all data.

### Terminology drift and a published erratum exist

CONCEPT.md uses 'pragmatism'; RELATIONS.md consistently uses 'pragmatics' as the field name (e.g. RELATIONS.md:34, 'pragmatics: identifiers'); semiotic-retrieval-research.md:76 uses SQL column 'pragmatism'. CONCEPT.md:43 and ref [2] note the 2018 published paper inadvertently swapped the semantics and pragmatism columns. Any datacrystal-level adoption must pick canonical field names; the concept itself treats them as synonyms.

### The relation model: 4 vocabulary layers + instances-vs-concepts split (RELATIONS.md)

Two entity kinds: instances (real invoices/emails, produced by SWARM pipelines) and concepts (SKOS vocabulary terms from TTL imports or Business Map boards). Four relation layers, each a distinct pragmatics value: (1) identifiers — properties uid (required, scoped URI 'sig://{vendor_slug}/{vendor_id}', the dedup key), primaryId, alsoKnownAs (borrowed from W3C DID); (2) owl/sameAs — bidirectional, idempotent instance-equivalence links written only by the automated dedup service when uids match; (3) skos/exactMatch — concept equivalence created by terminology import; (4) dcterms/subject — instance→concept classification written by SWARM and a retroactive linker. Claimed novelty: identity as a queryable pragmatics layer ('WHERE pragmatics = identifiers' without schema knowledge, RELATIONS.md:204-206). Relations are SDAs whose asset is the target URI — i.e. typed edges stored as ordinary records.

### Ontology and hierarchy live INSIDE the store as ordinary SDAs

SKOS concepts are stored as SDA rows (semiotic-retrieval-research.md:108-111): sigmatic 7KxmP2 with pragmatism=skos and semantics=label/definition/broader/synonym. Document structure is an 'outline hierarchy' via has_parents/has_children relations on SDA groups (semiotic-retrieval-research.md:89-99). So tree (PageIndex-style), entity graph (GraphRAG-style), and ontology hypergraph (OG-RAG-style) are all encodings within the one flat record model — the thesis of the planned 'semiotic retrieval' paper is that this unification is the contribution.

### GoBD demands: the storage-layer contract SDA must guarantee

gobd-ledger-research.md maps SDA to the four GoBD principles: Unveränderbarkeit (append-only + fingerprint), Vollständigkeit (sigmatics grouping + NEW Merkle layer — per-document hashes are insufficient, collection-level completeness proof is the identified gap, line 62), Nachvollziehbarkeit (editstamp + provenance/topology SDAs), Maschinelle Auswertbarkeit (syntax field + Z3 export: flat CSV + index.xml + DTD per Beschreibungsstandard v1.6). Architecture (§5): SQLite SDA store + Merkle audit layer (leaf = SDA fingerprint, root recomputed per emit, root-hash chain, O(log n) proofs, checkpoints optionally signed via RFC 3161/eIDAS timestamps) + optional WORM blob backend. Retention: timers derived from pragmatism (invoice→8y, annual-statement→10y, correspondence→6y per BEG IV), legal-hold flag, expiry deletion requires audited tree rebuild. IDW PS 880 certification is a medium-term goal.

### Retrieval semantics: orthogonal 4-axis queries + three-layer search architecture

Queries are conjunctive equality filters over categorical axes plus sigmatics lookup (semiotic-retrieval-research.md:74-83, literal SQL: WHERE pragmatism='object' AND semantics='classification' AND syntax='json'; WHERE sigmatics='X' ORDER BY pragmatism, semantics). Layered search (§3.4): Layer 1 symbolic = FTS5 + SKOS synonym expansion; Layer 2 reasoning = LLM agent with three tools (SPARQL over in-memory RDFLib concept graph, outline navigator over has_parents/has_children, FTS5); Layer 3 embedding = 'Triple Sigmatics'. Evaluation plan targets LongMemEval (>86%) and FinanceBench (vs PageIndex 98.7%) with ablations vs naive RAG/GraphRAG/PageIndex.

### Proposed model extensions: authority field and Triple Sigmatics

OUTLOOK.md Direction A: optional 8th tuple element 'authority' — an integer (0–1000); when multiple SDAs share sigmatics+pragmatism+semantics, highest authority is the effective value at query time, lower ones persist but are shadowed (CRDT-friendly conflict resolution + GoBD-compatible audit: 'LLM generated X (200) → human corrected to Y (700)'). authority=0 marks volatile, GC-able data exempt from retention. The org-specific level mapping is explicitly 'configuration, not part of the SDA model' (OUTLOOK.md:245). Open question whether authority is core tuple or annotation SDA. Direction B: sigmatics becomes (id, embedding_semantic, embedding_asset) — two vectors per record (~12KB at 1536 dims), embedding_asset recomputed on new content versions, old vectors retained with model-version tags for temporal similarity queries. CRDT-as-document-model (ops-as-SDAs with editstamp causal ordering) is also sketched but unprototyped.

### RDF is a projection, not the substrate — 'Approach C hybrid' is the stated recommendation

gobd-ledger-research.md:276-283 evaluates three options and recommends C: SDA store as primary storage, RDF graph (via URI sigmatics, P2P-O ontology for invoices) as query/linking layer, SPARQL without full data transformation. OUTLOOK.md drafts a minimal sda: RDFS/OWL-Lite vocabulary (sda:SDA class + 7 datatype properties) deliberately without inference requirements. CONCEPT.md:364-369 lists no-native-inference, no recursive graph traversal ('flat queries — topology must be processed'), and schema-freedom as accepted limitations.

### Timeless concept vs implementation detail

Timeless (the publishable model): the 7-tuple and semiotic axis definitions; immutability + emit/revoke; multi-faceted sigmatics identity; recursive decomposition; orthogonal querying; URI-compatible identity; instances-vs-concepts distinction; relation layers reusing standard vocabularies (owl/skos/dcterms/DID terms); fingerprint+editstamp as first-class integrity/temporality. Implementation detail / contingent: SQLite backing, '.sdaout' corpus format (CONCEPT.md:279), FTS5, RDFLib in-memory graph, the 'sig://{vendor}/{id}' uid pattern, SWARM pipeline ownership rules, Merkle tree as SQLite adjacency list, Z3/Beschreibungsstandard export, authority mapping tables, retention period numbers, spike numbers (#228, #286). Unsettled model questions: authority as 8th field, pragmatism-vs-pragmatics naming, Triple Sigmatics, CRDT extension. The implementation dir (/Users/sh/tetrahedron/projects/semantic-studio/src/semantic_studio/sda_store/: core.py, blob.py, tenant.py, plugins/, realm_import.py) confirms multi-tenancy and blob handling are real requirements though barely mentioned in the concept docs.

### Strategic context: SDA is an academic/publishable concept with a closing window

OUTLOOK.md defines a 5-paper pipeline: Paper 1 formal model (foundation, 'HIGH priority'), Paper 2 GoBD ledger (with co-author Björn Müller, audit-domain credibility), Paper 3 SDA↔RDF mapping, Paper R semiotic retrieval ('HIGH — competitive window closing', position paper targeted at SEMANTiCS 2026, deadline ~May 2026 — i.e. essentially now relative to 2026-06-10), Paper 4 LLM extraction. This matters for the build-on vs build-in question: the SDA model's identity as a citable, formally defined concept argues for keeping it a named layer with its own invariants rather than dissolving it into a general-purpose database's feature set.

## Relevance to datacrystal

GENERIC persistence primitives (storage-layer concerns SDA needs, datacrystal-shaped): (1) The SDA tuple is ONE fixed-schema record type — in datacrystal it is a single msgspec-record entity (7 fields: 4 short strings, 1 bytes/str payload, 1 hash, 1 int timestamp), not a new storage paradigm; sigmatics maps onto OID/secondary-unique-key lookup. (2) Orthogonal querying over pragmatism/semantics/syntax is filtering on low-cardinality categorical strings — a near-perfect fit for pyroaring bitmap indexes + the Condition AST; 'all facets of sigmatics X ordered by axes' is a bitmap-AND plus sort. (3) Layer-1 symbolic search = SQLite FTS5 sidecar (already planned); Layer-3 Triple Sigmatics = usearch vector sidecar, but requires TWO vectors per record plus model-version-tagged historical vectors — a multi-vector-sidecar generalization of datacrystal's current design, fed naturally by the idempotent commit-delta/watermark pipeline (embedding_asset recomputed exactly when a new SDA version for a sigmatics lands in a commit delta). (4) owl:sameAs / dcterms:subject / has_parents-has_children edges map onto the v1 reverse-reference index + traversal API (SDA stores edges as rows with URI assets — datacrystal could store them as typed references and get reverse traversal for free, fixing SDA's admitted 'flat queries, no recursive traversal' limitation). (5) The RDF 'Approach C hybrid' (store primary, RDF projection for SPARQL) is exactly datacrystal's planned RDF/triples extension package — SDA's sda: vocabulary is a ready-made projection target. (6) Z3 CSV export, pivot tables, and 'Maschinelle Auswertbarkeit' map onto Arrow columnar mirrors (DuckDB/polars). (7) Datomic-style append-only semantics: SDA's emit/revoke + editstamp resembles datacrystal's immutable watermark snapshots, but SDA demands it as the WRITE model, not just the read model — datacrystal's tri-state dirty tracking / mutable entities would need an 'append-only collection' mode where update is forbidden and revoke is a record. (8) Fingerprint-at-commit and a Merkle/hash-chained commit log (root hash per commit, O(log n) inclusion proofs, completeness proof between watermarks) is a genuinely generic primitive candidate — it slots onto the watermark/commit-delta pipeline and would differentiate datacrystal, but is NOT currently in its design. (9) tenant.py in the existing sda_store hints at multi-tenancy; datacrystal's embedded local-first persona handles this as DB-per-tenant at the application layer. DOMAIN/application semantics (belong ON TOP of datacrystal, in an SDA package): the semiotic INTERPRETATION of the four axes and their vocabularies; the identifiers/owl/skos/dcterms layer conventions and ownership rules (dedup service writes sameAs, importers write exactMatch); the dedup service, terminology importer, retroactive linker, SWARM extraction pipelines; the LLM reasoning search agent; authority-level mapping tables (explicitly 'configuration, not model'); GoBD specifics — retention-period policy per pragmatism, legal hold, Z3/Beschreibungsstandard export, Verfahrensdokumentation templates, IDW PS 880; P2P-O invoice ontology. BORDERLINE (could go either way): query-time authority shadowing (max-wins projection — generic 'effective view over event log' vs app logic); revoke-aware current-state projection (generic event-sourcing read-model machinery); recursive-decomposition lifecycle (app workflow, but benefits from generic lazy loading via Lazy[T] for unparsed assets/blobs). Net shape: SDA-as-concept is an application-level data model + vocabulary discipline that needs from its storage layer exactly four things datacrystal doesn't yet promise — enforced append-only collections, content fingerprints at commit, a verifiable hash-chain/Merkle audit primitive, and multi-vector sidecars — while everything else SDA queries on is already on datacrystal's roadmap (bitmap indexes, FTS5/usearch sidecars, reverse references, Arrow mirrors, RDF extension, snapshots).
