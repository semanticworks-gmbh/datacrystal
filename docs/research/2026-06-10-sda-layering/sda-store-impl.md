# SDA reading: sda-store-impl

_SDA layering study, 2026-06-10. Sources: /Users/sh/tetrahedron (concept docs, sda_store implementation, specs)._

## Summary

SDA store is an append-only EAV/quad store on SQLite (via SQLModel/SQLAlchemy async + aiosqlite) where every fact is one row in a single table `sda_store`: (sigmatic, pragmatics, semantics, syntax, asset, ingested_at). `sigmatic` is a URI identity (36-bit-random base58 base, dotted suffixes for hierarchy), `pragmatics` is a layer/namespace ('outline', 'business-map', 'identifiers', 'de.semanticworks.invoice'), `semantics` is the attribute name, `syntax` is a type tag ('txt','uri','json','outline','blob_ref','datetime'), and `asset` is the value, always normalized to a JSON-list string with cardinality (0-n/1-1/0-1/1-n) validated only in Python. Updates never mutate: `update_sda` simply inserts a new row and "latest version" is recomputed at read time via GROUP BY/MAX(ingested_at) self-joins. A plugin system injects SQL views; the deduplication plugin builds owl:sameAs/alsoKnownAs clusters with a recursive CTE and exposes "virtual SDAs" (redirects, cluster TOCs) through a `sda_union` view that ALL reads go through. Multi-tenancy is DB-per-tenant (`data/{org_id}/studio.db`) via a TenantRegistry with per-tenant asyncio locks; a co-located `blob_store` table holds binaries content-addressed by SHA-256, linked back as `syntax='blob_ref'` SDAs. Objects (outlines, boards) are reconstructed in Python by grouping rows per sigmatic and recursively following `has_children:uri` JSON arrays; a separate `search_index` + FTS5 table is rebuilt from scratch by walking every outline tree. The slowness is structural: every exact-URI read recomputes the entire dedup connected-components graph, every "latest" read GROUP-BYs over an ever-growing version history including the full asset text, and reads of one logical object fan out into N+1 per-node queries with per-row json.loads + Pydantic model construction.

## Key findings

### Schema: single EAV table + blob store + 5 views

core.py:301-340 defines `sda_store(id PK, sigmatic TEXT, pragmatics TEXT, semantics TEXT, syntax TEXT, asset TEXT, ingested_at DATETIME)` plus a `volatile INTEGER DEFAULT 0` column added by the dedup plugin (views.py:153-156). Indexes: idx_sda_sigmatic, idx_sda_sigmatic_pragmatics_semantics, idx_sda_ingested_at (core.py:520-538). blob.py:77-116 creates `blob_store(id, sigmatic, mime_type, filename, data BLOB, created_at, content_hash)` with indexes on sigmatic and content_hash. The dedup plugin adds views v_alt_pairs, v_clusters, v_object_toc, v_redirects, sda_union (views.py). Asset values are ALWAYS JSON-list strings (normalize_asset, core.py:184-213); cardinality (Cardinality enum core.py:172-181) is enforced only by an optional Python validator, never by the DB.

### Operations API: CRUD-by-append, prefix queries, no real relation traversal

SDAService (core.py:417+) offers create_sda (insert), update_sda == create_sda (core.py:776-791, versions accumulate forever, only unpublish_outline at core.py:1021 ever deletes), get_sdas_by_uri / get_sdas_by_uris_batch (IN-clause batch, core.py:601-664), get_sdas_by_uri_prefix (LIKE 'prefix%'), get_latest_by_sigmatic, get_version_history, raw execute_query. Hierarchy = URI string convention: get_hierarchy_descendants is a LIKE 'base.%' query (core.py:793-797); siblings are filtered in Python by string depth (core.py:799-818). Relations are NOT first-class: create_relation (core.py:851-864) writes an SDA with pragmatics='relation' and asset string 'from_rel_to'; get_related_sdas (core.py:831-849) does Python substring containment `sigmatic_uri in sda.asset` — reverse-reference lookup ('who points at me') has no index and degrades to scans. Graph edges in practice are `has_children:uri` / `has_parents:uri` JSON arrays inside asset.

### Plugin architecture: SQL-view injection + named capability hooks

plugins/base.py defines SDAPlugin ABC with hooks: on_start(service), provide_views() -> list of CREATE VIEW DDL, extend_queries(query_context) (defined but unused anywhere), refresh_views, get_status, and dedup-specific capability methods get_object_versions / get_canonical_sigmatic / get_cluster_members that the core service dispatches to by plugin name (core.py:880-897 raises NotImplementedError without the plugin). SDAPluginRegistry is a class-level global list with priority ordering (base.py:110-154); plugins auto-register at import time (__init__.py:39). So the only real extension mechanism that ships is: a plugin can splice SQL views into the read path — which the dedup plugin uses to redefine what ALL reads see.

### Multi-tenancy: DB-per-tenant with lazy registry

tenant.py: TenantRegistry maps org_id -> SDAService, each tenant gets `data/{org_id}/studio.db` (tenant.py:121-123) with its own AsyncEngine; default tenant '_default'. get_or_create (tenant.py:75-110) lazily mkdirs, creates engine, runs create_tables (idempotent schema + plugin view bootstrap) under a per-tenant asyncio.Lock with double-checked caching. get() is sync cache lookup that raises if uninitialised. BlobService is co-located on the same engine per tenant (blob.py:31-47). db_setup.setup_all_tenants iterates data/*/studio.db dirs. Clean, simple isolation model — no cross-tenant queries exist or are possible.

### Slowness root cause #1: every read recomputes dedup clusters via recursive CTE

core.py:455-464 `_get_read_table` routes ALL read queries through the `sda_union` view. sda_union (views.py:140-150) UNION-ALLs sda_store with v_object_toc and v_redirects, both built on v_clusters (views.py:21-66) — a WITH RECURSIVE flood-fill connected-components computation over v_alt_pairs, which itself runs json_each() over every `pragmatics='identifiers' AND semantics='alsoKnownAs'` asset (views.py:10-18). SQLite recomputes this entire graph clustering on EVERY query, even `get_sdas_by_uri('X')` for one URI — predicate pushdown cannot shortcut a recursive CTE or the json_group_array aggregations in v_object_toc. Cost grows superlinearly with the alsoKnownAs graph and linearly with table size for every single read. This is the dominant 'slow when growing' mechanism.

### Slowness root cause #2: latest-version dedup is a per-read GROUP BY over full history including the asset text

Because updates append rows forever, 'latest' is computed at read time by a self-join: GROUP BY sigmatic, pragmatics, semantics, syntax, asset with MAX(ingested_at), joined back on all five columns including equality on the full JSON `asset` text (core.py:467-484, 494-511, 631-646). Grouping/joining on an unbounded TEXT column is unindexable through the sda_union view and scans all versions of all matching rows. Every edit to an entity makes every future read of it slower; nothing ever compacts old versions. There is no per-version-tuple index, and the GROUP BY happens over the view, so the sda_store indexes (core.py:520-538) only help the inner WHERE.

### Slowness root cause #3: N+1 object reconstruction in Python

A logical object is K rows; reading it means fetching rows then json.loads per row (get_asset_values core.py:216-231) plus SDA(**row) Pydantic construction per row with a model_fields filter dict-comp (core.py:683-685 and 5 more copies). Tree builds are recursive one-query-per-node: render_processor/outline_tree.py:16-91 (build_outline_tree calls get_sdas_by_uri per child); the batch variant (outline_tree.py:94-197, one IN-query per BFS level) exists but many hot paths still loop: search/index.py:384 and :619 (per child_uri / per parent in breadcrumb walk), search/article_detail.py:61+90 (parent-chain walk, one query per ancestor), terminology/board_enrichment.py:95 (per board element), dedup/service.py:472. blob.py:275-283 _ensure_blob_sda fetches all SDAs for a sigmatic and scans them in Python just to check existence.

### Slowness root cause #4: rebuild-the-world maintenance jobs

db_setup.setup_database (db_setup.py:270-284) runs batch dedup AND a full search-index rebuild on every setup. SearchIndexService.rebuild (search/index.py:136-198) discovers all roots, walks every outline tree (re-querying SDA store throughout), calls get_max_ingested_at_by_sigmatic — a full-table GROUP BY materialized into a Python dict for ALL sigmatics (core.py:900-922) — then DELETEs the whole search_index and re-inserts row-by-row via ORM session.add, then issues FTS5 'rebuild' which re-reads everything (index.py:117-130). import_json_data (core.py:931-975) creates one AsyncSession + one COMMIT per attribute row — per-row fsync makes bulk import O(rows) transactions. discover_root_outlines (outline_tree.py:307-341) selects ALL pragmatics='outline' rows and computes roots in Python; get_published_outlines (core.py:1047-1066) filters on pragmatics+semantics which no index leads with, scanning sda_union (including the recursive CTE).

### Versioning & dedup semantics worth keeping

Conceptually strong pieces: (a) append-only versioning with ingested_at and a version-tuple identity — time travel for free; (b) content-addressed blob store (SHA-256, idempotent upsert, hash-verified reads, blob_ref SDAs placing binaries inside the semantic model, blob.py:118-218); (c) entity dedup as data, not destruction: identifiers/uid -> owl:sameAs links stored as ordinary SDAs, canonical election by oldest-wins with manual override (dedup/service.py docstring lines 1-52), and 'virtual SDAs' (redirects + cluster TOC) that make merged identities transparently readable at original URIs; (d) the batch query API get_sdas_by_uris_batch explicitly built to kill N+1 (core.py:601-664); (e) idempotent, re-runnable migrations (db_setup.py:32-178).

### The four-layer semantic model is the actual invention

The durable idea is the addressing scheme: sigmatic (identity URI) x pragmatics (layer/context namespace) x semantics (attribute) x syntax (type tag) -> asset (value list with cardinality). It lets multiple producers (REALM import, SWARM sync, SKOS terminology enrichment, blob ingest, manual edits) write into different layers of the SAME identity without coordinating schemas — realm_import.py merges 'outline' + 'business-map' layers per sigmatic (realm_import.py:91-132); board_enrichment adds a SKOS layer post-hoc. The 'semantics:syntax' key convention in import JSON (core.py:954-962, e.g. 'has_children:uri', 'title:txt') is a compact serialization of typed attributes. This layered-namespace-on-one-identity model is genuinely generic; everything else (outline trees, boards, publish flags, invoice heuristics in blob.py:220-244) is application vocabulary.

## Relevance to datacrystal

GENERIC persistence primitives (belong in/under datacrystal): identity URIs (sigmatic ≈ datacrystal OID registry — base58 random base + hierarchical child suffixes maps to OIDs + owned sub-entities); append-only versioning + latest-view (maps to datacrystal's immutable watermark snapshots — 'latest' should be a maintained state, not a per-read GROUP BY); content-addressed blob store (a natural datacrystal sidecar alongside FTS/vector); DB-per-tenant registry (maps to one datacrystal store per directory; the TenantRegistry pattern survives as app code opening multiple stores); batch fetch API (subsumed by datacrystal entity loads + Lazy[T] with reverse-reference index); search_index/FTS5 rebuild pipeline (exactly what datacrystal's idempotent commit-delta/watermark pipeline replaces — incremental instead of delete-all+rebuild); the unused extend_queries hook is what Condition AST queries actually deliver. DOMAIN/application semantics (stay on top): pragmatics layer vocabulary ('outline', 'business-map', 'PBI', invoice mime-type heuristics blob.py:220-244), REALM board validation/slugs, SKOS enrichment, publish/unpublish, outline tree rendering. MAPPING PER SDA ELEMENT: the (sigmatic, pragmatics, semantics, syntax, asset) quad is isomorphic to datacrystal's planned RDF/triples extension (sigmatic=subject, pragmatics+semantics=predicate-with-namespace, syntax=datatype tag, asset=object list) — SDA-as-schema-free-layers fits the RDF extension package, while stable layers (outline nodes, boards) should become typed slots-dataclass entities with cardinality expressed as field types (1-1 → T, 0-1 → T|None, 0-n → list[T]), has_children:uri → list[Lazy[Node]] resolved via the v1 reverse-reference index (killing the Python substring scans in core.py:845-849 and the parent-chain N+1 walks); dedup clusters → application logic over bitmap indexes + reverse-reference index, with connected components maintained incrementally at write time (union-find on commit) instead of the per-read recursive CTE; v_redirects/v_object_toc 'virtual SDAs' → derived entities refreshed by the watermark pipeline; get_max_ingested_at_by_sigmatic / table_export → Arrow columnar mirrors (DuckDB/polars) instead of full-table GROUP BY into Python dicts; search_index_fts → datacrystal's FTS5 sidecar fed by commit deltas. VERDICT IMPLICATION FROM EVIDENCE: every identified slowness cause (per-read recursive CTE, per-read version GROUP BY, N+1 reconstruction, rebuild-the-world indexing, per-row commits) is a symptom of building persistence machinery (versioning, dedup views, latest-resolution, search feeding) inside application SQL — these are precisely the primitives datacrystal already plans (snapshots, indexes, watermark pipeline, reverse refs). The SDA four-layer addressing model is a data-modeling vocabulary, not a storage engine: it fits cleanly as an extension package (like the planned RDF extension, possibly the same one) plus typed entities for hot paths, NOT as a change to datacrystal's core concept.
