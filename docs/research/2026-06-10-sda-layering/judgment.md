# SDA layering judgment

_2026-06-10. Decision support for: does SDA belong in datacrystal core, or on top?_

## Verdict

SDA goes ON TOP of datacrystal — as a named application-layer package (the SDA model/vocabulary) plus app code in semantic-studio — with a principled, narrow back-flow of exactly four generic requirements into datacrystal (unique secondary-key lookup, frozen/append-only entity mode, earlier FTS sidecar, a future ledger extension on the watermark pipeline). It must NOT become part of datacrystal's core concept. The argument: every major SDA element fails at least one layering test for core inclusion — the semiotic axes, relation-layer vocabularies, dedup policy, authority shadowing, and GoBD compliance are domain knowledge the database cannot have (test 2) and that no non-SDA user wants (test 1); meanwhile everything SDA needs from storage is either already on the ratified roadmap (bitmap indexes + Condition AST, FTS5/usearch sidecars on the commit-delta/watermark pipeline, reverse-reference index, snapshots, Arrow mirrors, RDF extension) or is a one-flag generalization of it. The evidence from sda_store is decisive: all five measured slowness causes (per-read recursive-CTE dedup clustering, latest-version GROUP BY over full asset text, N+1 Python reconstruction, rebuild-the-world FTS indexing, per-row commits) are symptoms of hand-rolling persistence primitives in application SQL — i.e., SDA is the proof-by-counterexample for datacrystal's existence, not a missing core feature. Strategically, SDA's identity as a citable formal model (5-paper pipeline, SEMANTiCS window) is better served by 'SDA: a semiotic data model, reference implementation on datacrystal' than by dissolving into a database's feature list; and the solo-maintainer/24-month constraint ratified in ROADMAP.md cannot absorb a second conceptual universe in core. Datacrystal's typed slots-dataclass identity and SDA's schema-free EAV are philosophically opposed — the synthesis is typed entities for SDA's stable layers (outlines, concepts, blobs) and a generic frozen Statement entity for the schema-free long tail, both as ordinary datacrystal application classes.

## What datacrystal core gains from SDA as first customer

SDA as first customer contributes: (1) PROMOTE unique secondary-key index (string alias -> entity) from implied EntityMap detail to an explicit v0.x commitment — sigmatic-URI lookup is SDA's hottest read path and idempotent upsert-by-uid is its write-path idempotency mechanism; generic for any app with natural keys (test 3: everyone rebuilds this badly). (2) ADD frozen/append-only entity mode (@entity(frozen=True) or append_only collection: update raises, delete forbidden outside GC policy) — near-zero cost given tri-state dirty tracking (entities never arm Dirty), and generic for the named agent-memory persona (episodic logs, transcripts, provenance) — test 1 and 3 both pass; this is the only SDA invariant worth enforcing in core, and it benefits everyone without constraining mutable entities (test 4 passes because it is opt-in per class). (3) RESEQUENCE datacrystal[fts] earlier — the ratified roadmap defers all extensions until 'after v1 core freeze', but roadmap item 3 calls the commit-delta/watermark pipeline 'the single most load-bearing undelivered component'; a public contract with zero consumers is untested. FTS5 is the cheapest consumer (stdlib) and SDA needs it; pull it back to late v0.x as the pipeline's validation harness (DESIGN.md's original v0.2 had exactly this). (4) ADD to the punt list: datacrystal[ledger] — hash-chained commit log + Merkle inclusion/completeness proofs as a Tier-3 watermark consumer; demand-driven, differentiating for audit and agent-provenance use cases; its only requirement on core is deterministic, stably-ordered, replayable commit deltas — already promised in item 3, so SDA's GoBD requirement becomes a free conformance test of that promise. (5) VALIDATE the asyncio story early: semantic-studio is fully async (FastAPI+NiceGUI), so aopen() loop binding, store.transaction(), and the 'critical section is the code between awaits' doctrine get a real customer in v0.x rather than at the v1 web extension. (6) ADD a batch-hydration API (load-many-by-OID) — the get_sdas_by_uris_batch lesson; trivial, generic, prevents N+1 from ever being the user's problem. (7) CONFIRM multi-vector commitment in docs (>=2 @Vector fields per entity for Triple Sigmatics). DEMOTIONS/non-gains: SDA does not justify accelerating Arrow mirrors (hundreds-to-thousands of rows per tenant; v1 timing fine), does not need the reverse-reference index before v1 (it already stores both edge directions explicitly), does not move the RDF extension off the punt list (Wave-3 SPARQL runs over in-memory RDFLib today and can keep doing so at app level), and adds zero pressure toward any 'Never' item.

## What must NOT leak into core

(1) Schema-free EAV/statement storage as a core mode — it destroys datacrystal's identity (typed slots-dataclasses AS the schema), violates the hard 'never expose object-dtype columns' rule for Arrow mirrors, and degrades bitmap indexing to string-soup; if a generic Statement/quad primitive is ever wanted, it is the datacrystal-rdf extension's job, not core's. (2) Semiotic axis names (sigmatics/pragmatics/semantics/syntax) anywhere in core API or docs — SDA's own corpus can't keep pragmatism-vs-pragmatics straight (published erratum); a database must not carry a contested domain vocabulary. (3) GoBD/compliance semantics in core (retention periods, legal hold, WORM, Z3 export, IDW PS 880 certification posture) — regulatory domain knowledge with audit liability, fatal to a solo maintainer; test 2 failure in its purest form. (4) Merkle-root computation in the core commit path — taxes every commit of every user for a compliance feature; the watermark pipeline makes it a lossless sidecar, so in-core placement buys nothing (test 4 failure: constrains everyone, benefits only SDA). (5) Dedup/sameAs/canonical-election/redirect resolution in the core read path — this is the sda_union mistake (per-read recomputation of domain policy) recreated; core reads must stay O(lookup), projections are app-maintained derived data. Oldest-wins election and manual override are policy, not storage. (6) Authority-based conflict shadowing in core — it is LWW-with-authority application policy; the roadmap's 'Never' list already rejected richer-than-LWW conflict semantics when it rejected core CRDTs. (7) Multi-tenancy registry in core — multi-store-per-tenant is already the answer; a tenant concept in core duplicates the filesystem. (8) 'Current-state projection' / materialized-view DSL in core v0.x — tempting generalization of SDA's latest-wins need, but it is a query-engine commitment; the watermark-consumer contract is the principled seam, and app code owns projections until >=2 more customers demand otherwise. Each of these would also break the ratified scope discipline: core caps at two sidecar technologies, ~24-month v1.0, solo maintainer.

## Migration sketch (sda_store → datacrystal)

Rebuild as 'sda' package over datacrystal (gated on core v0.x items 1-4 plus the FTS extension — /Users/sh/pyrsistance is design-docs-only today, so this is the first-customer workload, not a this-month port). ENTITIES (typed, for stable layers): OutlineNode(sigmatic: Annotated[str, pyr.Id-unique], title, description: Annotated[str, pyr.FullText], children: list[Lazy[OutlineNode]], parents: list[Lazy[OutlineNode]], published: datetime|None, pbi_status, needs_review); BoardElement; SkosConcept(label, definition, broader: list[Lazy[SkosConcept]], synonyms: list[str], exact_match: list[Lazy]); Blob(sha256 unique, mime, filename, data: Lazy bytes); DedupCluster(canonical: Lazy, members: list[Lazy], elected_at) + Redirect entities. SCHEMA-FREE TAIL: one frozen Statement entity (@entity(frozen=True)) carrying the 5-tuple (sigmatic, pragmatics, semantics, syntax, asset_json) + fingerprint + editstamp, with pyr.Index bitmap markers on the three categorical axes — emit = store(), revoke = a frozen revocation Statement; this preserves SDA's append-only event-log semantics and audit history verbatim while typed entities serve as the maintained 'latest' projection updated in the same commit. INDEX/SIDECAR MAPPING: sigmatic lookup -> unique identity index; all axis scans -> pyroaring bitmaps + Condition AST; search_index+FTS5 rebuild machinery -> deleted, replaced by datacrystal[fts] sidecar fed O(delta) by commit watermarks (breadcrumbs become a denormalized field maintained at write); Wave-4 embeddings -> datacrystal[vector], two @Vector fields; sda_union/v_clusters views -> deleted, dedup runs as incremental union-find at write time (commit-delta consumer) materializing DedupCluster/Redirect entities; table_export -> Arrow mirror + DuckDB when v1 lands; SPARQL Wave 3 -> keep in-memory RDFLib projection in-app now, adopt datacrystal-rdf extension if/when it ships; GoBD Merkle layer -> datacrystal[ledger] watermark consumer. TENANCY: TenantRegistry survives shape-identical, opening one store per org dir (data/{org}/studio.dc) via aopen() on the event loop; per-tenant asyncio.Lock replaced by ADR-001 confinement + store.transaction(). WHAT FIXES THE SLOWNESS — structurally fixed, not moved: per-read recursive CTE -> moved to write time and made incremental (the correct side; cost now proportional to new sameAs edges, not to corpus x reads); latest-version GROUP BY over asset text -> gone, 'current' is the live entity, O(1) identity-map/OID hit; N+1 + per-row json.loads/Pydantic -> gone, msgspec decode ~93ns/obj, identity-map caching, Lazy batch hydration, real references instead of string joins; full search rebuild per write batch -> O(delta) watermark application; one-commit-per-row imports -> buffer-until-commit storer with group-commit fsync. WHAT REMAINS/MOVES: write-time dedup maintenance is app code that must be written and tested (complexity relocates from opaque SQL views to debuggable Python — a win, but not free); Python object memory (~600B/obj) and 18-26ns/obj scan floors remain, irrelevant at SDA scale (10^2-10^5 rows/tenant vs the 1-5M envelope). STAYS IN SEMANTIC-STUDIO: FastAPI/NiceGUI UI, render processors, SWARM push contract (idempotency simplifies to upsert-by-uid), SKOS/TTL import, synonym-expansion direction policy, publish/unpublish workflow, dedup matching rules + election policy, retention/Z3/GoBD application layer, LLM search agent, Business-Map extraction. INTERIM (if pain bites before datacrystal v0.2-equivalent): materialize v_clusters into a table refreshed on dedup writes and add a latest-pointer table — two-day patches to sda_store that mirror the eventual architecture.

## Dissent (the case for SDA-in-core, and why it loses)

Strongest case for SDA-IN-CORE: the SDA 5-tuple IS the datom/quad primitive datacrystal already half-plans (the RDF extension), and the agent-memory persona — datacrystal's flagship — natively emits schema-free facts (LLM-extracted triples, observations, provenance). Fusing them would make datacrystal 'the semantic object database': a native Statement store with axis bitmaps, append-only semantics, fingerprints, and a Merkle-verifiable log as THE differentiator no embedded Python DB has, with SDA as the published formal model giving it academic legitimacy (Paper 1 becomes the database's theory paper). It loses on four grounds: (a) the ratified roadmap explicitly relegates triples to an extension (punt #17) and caps core at two sidecar technologies after a 14-agent study plus adversarial stress test — reversing that for the first customer is exactly the one-customer-driven scope creep the process exists to prevent; (b) schema-free EAV contradicts the load-bearing walls — typed dataclasses-as-schema, typed-column Arrow mirrors, the no-object-dtype rule — so fusion weakens both identities rather than strengthening either; (c) solo maintainer, 24-month honest v1.0: a second core data model roughly doubles format-freeze surface, migration surface, and docs; (d) sda_store's measured pathologies came precisely from entangling domain semantics (dedup views, latest-wins, virtual SDAs) with the storage read path — the lesson of the existing implementation is separation. SCENARIO WHERE IT WINS: if, post-v1, agent-memory demand converges on untyped fact emission as the dominant workload (multiple real customers asking for a statements API, not just SDA), then the SDA tuple should be ratified as the canonical statement format OF the datacrystal-rdf/statements extension — its spec, vocabulary-neutral, with frozen-entity enforcement and fingerprints — i.e., SDA-the-tuple graduates into the extension layer, still never into core.
