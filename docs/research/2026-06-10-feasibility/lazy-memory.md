# Finding: lazy-memory

_Feasibility study, 2026-06-10. One of 10 parallel investigations (5 reading the EclipseStore source in `resources/store`, 5 researching the Python ecosystem)._

## Summary

EclipseStore does partial graph loading WITHOUT owning the heap via three cooperating pieces: (1) Lazy<T> — a plain, explicit wrapper object (no proxies/bytecode magic) holding {subject strong-ref, objectId, transient ObjectSwizzling loader, lastTouched}; clear() nulls only the subject and keeps the OID. (2) LazyReferenceManager — a global singleton daemon thread (1s interval, 1ms time budget per cycle = 0.1% duty) that weakly tracks every Lazy and clears them via Lazy.Checker (default timeout 1,000,000 ms ≈ 15 min, memory quota default 1.0, age-weighted memory-pressure formula, heap stats sampled once per cycle). (3) PersistenceObjectRegistry (the swizzle registry) — a bijective OID<->object map whose entries ARE WeakReferences in two parallel hash tables; on Lazy.get() after clear, PersistenceManager.getObject(oid) first does a registry lookup (zero I/O if the instance is still heap-live) and only then loads binary from storage and re-registers. The JVM GC plays four roles: it is the ONLY thing that actually frees memory (clear() just severs a strong ref), it clears registry WeakReferences (ReferenceQueue drained lazily), its collections feed back into the on-disk storage GC (registry live-OID set is both sweep keep-alive predicate and mark-root seed), and the clear->collect window gives free identity-preserving re-gets. In CPython nothing is fundamentally impossible; refcounting makes the registry EASIER and more deterministic (weakref.WeakValueDictionary with immediate callback removal eliminates Java's cleared-but-unreaped ReferenceQueue window that EclipseStore needs elaborate defenses for); the two real CPython frictions are that core builtins (list/dict/tuple/int/str) are not weakreferenceable and that cyclic graphs defer collection to the gc module.

## Key findings

### Lazy<T> is a dumb explicit wrapper, deliberately no proxy magic

org.eclipse.serializer.reference.Lazy.Default fields: T subject (the only strong ref), long objectId (negative = unpersisted), transient ObjectSwizzling loader, long lastTouched. get() is synchronized: if subject==null && Swizzling.isProperId(objectId) -> load() -> loader.getObject(objectId); then touch() (lastTouched = subject!=null ? currentTimeMillis : Long.MAX_VALUE). clear() throws IllegalStateException if !isStored() (would lose data), otherwise nulls subject keeping objectId. peek() returns subject without loading. Null-safe statics Lazy.get/peek/clear(Lazy). Docs (resources/store/docs/.../lazy-loading/index.adoc) explicitly justify rejecting @Lazy-annotation/bytecode/proxy approaches (Hibernate-style) as undebuggable; the visible wrapper type is the design. Every Lazy self-registers with the global manager via static Lazy.register() called from factories.

### LazyReferenceManager: weak tracking, budgeted background sweep, Controller-gated

Global singleton (static globalReferenceManager). Registered lazies are kept in a linked list of Entry extends WeakReference<Lazy<?>> with O(1) tail append — the manager never keeps a Lazy (or its subject) alive. LazyReferenceCleanupThread is a daemon holding only a WeakReference to the manager (auto-terminates when manager is GC'd); DEFAULT_CHECK_INTERVAL_MS=1_000, DEFAULT_TIME_BUDGET_NS=1_000_000 — i.e. scan ~1ms per second with a cursor that resumes where it left off, reaping orphaned entries as it goes. Controller is a functional interface mayRun(); EmbeddedStorageManager.Default implements LazyReferenceManager.Controller (storage/embedded/.../EmbeddedStorageManager.java lines 148, 272-317, 458): starting a storage ensures the manager runs, shutdown() calls removeController(this) — clearing only happens while some storage is active.

### Lazy.Checker: timeout + memory-quota formula with age penalty

Defaults: timeout 1_000_000 ms (~15 min), memoryQuota 1.0 (=may use whole heap; 0.0 disables memory check). graceTime = min(1000ms, timeout/2): refs touched within grace are never cleared. Per cycle, beginCheckCycle() samples heap usage ONCE (MemoryStatisticsProvider.get().heapMemoryUsage()) and precomputes cycleTimeoutThresholdMs/cycleGraceTimeThresholdMs; sh10 = value<<10 fixed-point (~*1000). Decision per ref: age > timeout -> always clear; else clear if sh10MemoryUsed + cycleMemoryUsed*sh10Weight >= sh10MemoryLimit where sh10Weight = shift10(age)/timeoutMs — so under memory pressure older references are cleared first, under no pressure nothing is cleared before the hard timeout. Custom: LazyReferenceManager.set(LazyReferenceManager.New(Lazy.Checker(timeoutMs, quota))) before storage start.

### Reload path on Lazy.get() is registry-first, disk-second, identity-preserving

PersistenceManager.Default.getObject(long): `if((cachedInstance = this.objectManager.lookupObject(objectId)) != null) return cachedInstance; return this.createLoader().getObject(objectId);`. So clear() followed shortly by get() usually costs zero I/O — the instance is restored from the swizzle registry while the JVM GC hasn't collected it yet (docs/.../clearing-lazy-references.adoc states this explicitly). Only on a registry miss does a loader read the binary record(s) from storage channels, instantiate the sub-graph, and register each built instance under its OID, guaranteeing one live instance per OID (bijection) and referential identity across independent Lazy.get() calls.

### Swizzle registry = DefaultObjectRegistry: bijective OID<->object via dual hash tables of WeakReferences

Entry extends WeakReference<Object> { final long objectId; int refHash; Entry oidNext, refNext; } — the hash-table entries ARE the weak references. Two parallel arrays: oidHashTable keyed (int)objectId & hashRange, refHashTable keyed System.identityHashCode(object); the same Entry is chained in both, giving O(1) lookupObject(oid) (returns e.get()) and O(1) lookupObjectId(obj) (identity comparison e.get()==object). ReferenceQueue-based cleanUp() (polled on every storer merge via PersistenceObjectManager.synchInternalMergeEntries) plus full-scan consolidate() (called by StorageConnection#issueGarbageCollection). Default hash density 1.0f, range [0.75;2.0]. Registry never causes leaks: weak-only referencing means JVM GC collects app-dropped instances regardless of registration.

### JVM GC's precise role: it is the actual memory reclaimer and the bridge to the storage GC

Four roles. (1) Reclamation: Lazy.clear() and LazyReferenceManager only sever strong references; memory is freed only when the JVM GC collects the now only-weakly-reachable subject (docs say verbatim: 'That's the job of the garbage collector of the JVM'). (2) Registry hygiene: GC clears WeakReferences and enqueues them on the ReferenceQueue; EclipseStore drains lazily (cleanUp on store, consolidate on issued GC) — JDK semantics, the GC never removes entries from user containers. (3) Storage-GC feedback (storage/storage/GC.md, exceptional 440-line doc): the on-disk mark-and-sweep keeps an entity if isGcMarked() OR isReachableInApplication(oid), where the predicate is the registry's id-only synchContainsObjectId; the registry is also iterated to seed mark roots (enqueueLiveApplicationOids) so app-held subgraphs are walked transitively. Thus a JVM-GC collection of an instance (plus reaping) is what eventually allows the storage GC to delete its on-disk entity. (4) The clear->collect window enables free re-gets from the registry.

### The hard correctness problem is the Java 'cleared-but-unreaped' WeakReference window — which refcounting eliminates

GC.md sections 8-10 document the subtlest hazard: between JVM-GC clearing a WeakReference and the registry reaping its Entry, the registry contains entries whose get()==null (case (b)). The sweep keep-alive predicate and the mark-root seed must use the SAME id-only criterion (synchContainsObjectId, not synchContainsLiveObject), otherwise sweep keeps a parent whose binary references were never walked at mark time -> child entity deleted -> zombie OID -> StorageExceptionConsistency on reload (scenario spelled out step-by-step in GC.md section 9, involving BinaryStorer skipping re-serialization of already-registered objects). Three defenses: shared criterion, cleanUp() on every store, consolidate() on issued GC. This entire defense complex exists ONLY because Java's WeakReference clearing is asynchronous and queue-based.

### CPython analog: WeakValueDictionary is a near-exact and strictly simpler swizzle registry

OID->object: weakref.WeakValueDictionary[int, object] — entries are removed by weakref callback at the exact moment the last strong ref dies (refcounting), so 'contains oid' is always exact: no ReferenceQueue, no cleanUp/consolidate, no case-(b) window, no shared-criterion defense needed (modulo cyclic garbage, see below). Object->OID reverse map: Java needs the identityHashCode refHashTable because it cannot touch objects; Python can simply stamp the OID on the instance (obj.__pyrsistance_oid__ via __slots__ or instance dict) or keep dict[id(obj)->oid] guarded by weakref.finalize to invalidate id() reuse. Lazy[T]: trivial class with __slots__=(subject, object_id, loader, last_touched) + threading.Lock (needed for free-threaded 3.13+; under GIL the check-then-load must still be made atomic), typing.Generic[T], plus pydantic core-schema hooks so Lazy fields validate/serialize. Manager: daemon threading.Thread with 1s interval/1ms budget translates directly; offer an asyncio-task variant for FastAPI.

### Two genuine CPython gaps: non-weakreferenceable builtins, and cycles defer to gc module

(1) list, dict, tuple, set, int, str, frozen dataclasses without weakref_slot are NOT weakreferenceable (no __weakref__). Java can weakly reference anything. Consequence: the registry cannot weakly hold a raw list/dict subject. Mitigations: register only the Lazy-wrapped container or trivial subclasses (class PersistentList(list): __slots__=('__weakref__',) — subclassing adds weakref support), or keep strong refs for builtins inside the Lazy itself and rely on Lazy.clear() (the Lazy already owns the only strong ref to its subject, which is the EclipseStore model anyway — the registry weak-entry is an optimization, not a requirement, for leaf containers). dataclasses: weakref_slot=True exists since 3.11. (2) Reference cycles in the domain graph mean refcounting alone won't collect cleared subjects; collection waits for the generational gc — weakref callbacks still fire when gc collects a cycle, so correctness holds, but 'deterministic' freeing is only guaranteed for acyclic data. Optionally call gc.collect() from the checker when memory quota is exceeded, and use gc.callbacks/gc.freeze for tuning.

### Memory-pressure detection must be redesigned: no JVM heap quota in CPython

Java's Checker reads Runtime/MemoryUsage (used vs max heap) — meaningful because the JVM has a bounded heap (-Xmx). CPython has no max-heap concept; the analog is process RSS vs a user-configured budget: psutil.Process().memory_info().rss (or resource.getrusage(RUSAGE_SELF).ru_maxrss as a stdlib-only fallback, noting it is a high-water mark on Linux) sampled ONCE per check cycle exactly like beginCheckCycle(). Keep the same decision shape: grace time min(1s, timeout/2), hard timeout always clears, otherwise age-weighted pressure formula used + cycle_used*(age/timeout) >= budget. sys.getsizeof/tracemalloc are per-object/diagnostic, not suitable for the hot path. Also note RSS doesn't drop immediately after collection (allocator arenas/pymalloc), so the quota should gate clearing decisions, not be treated as a post-hoc success signal.

## Implications for the Python port

REPLICATE (the model maps cleanly): (1) Explicit Lazy[T] wrapper with .get()/.peek()/.clear() and module-level null-safe helpers — EclipseStore's anti-proxy argument applies equally in Python (debuggability, no __getattr__ magic); offer an optional transparent-proxy subclass later if demanded. (2) Touched-timestamp + global LazyReferenceManager singleton with daemon thread, 1s interval / ~1ms time budget, cursor-resumed incremental scan over weakly-held Lazy entries (weakref.ref to the Lazy objects themselves so the manager never pins graphs), Controller protocol tying it to StorageManager lifecycle (start on storage start, mayRun()=storage.is_running). (3) Checker with (timeout_ms, memory_quota) and the grace/timeout/age-penalty formula, swapping JVM heap stats for once-per-cycle psutil RSS vs configured budget. (4) Registry-first reload: PersistenceManager.get_object(oid) checks WeakValueDictionary before touching storage, then loader builds + registers, preserving one-instance-per-OID identity — this is what makes clear()+re-get() cheap and graphs referentially sound. (5) The storage-GC bridge invariant from GC.md: the on-disk mark-sweep's keep-alive predicate and its mark-root seed must use the same registry-membership criterion; in Python the registry is exact (refcount callbacks) so this collapses to 'snapshot registry keys, use snapshot for both seed and sweep' — vastly simpler, but keep the dual-role design (sweep filter = shallow safety net; mark seed = transitive walk of app-held subgraphs) because the Holder/Payload zombie scenario in GC.md section 9 is GC-strategy-independent and will bite any lazy-storer that skips re-serializing registered objects. SIMPLIFY: drop ReferenceQueue/cleanUp/consolidate machinery entirely (weakref callbacks remove entries synchronously); drop synchContainsObjectId-vs-containsLiveObject duality; the hot/cold two-phase seeding subtleties of GC.md sections 10.1/10.2 shrink because there is no cleared-but-unreaped window. DO DIFFERENTLY: (a) stamp OIDs onto instances (__pyrsistance_oid__) instead of an identity-hash reverse table — Python allows it, Java couldn't; fall back to id()-keyed dict + weakref.finalize for __slots__ classes you don't control; (b) handle non-weakreferenceable builtins by policy: the Lazy itself holds the only strong ref to its subject (so clear() works regardless), and registry weak-entries are only created for weakreferenceable objects — provide PersistentList/PersistentDict shims or recommend dataclass(weakref_slot=True) entities; (c) provide an asyncio-native manager variant (FastAPI) alongside the thread; (d) make Lazy.get() lock-protected for free-threaded CPython 3.13+; (e) when memory quota is breached, optionally trigger gc.collect() after a clearing pass so cyclic subgraphs actually die. VERDICT: nothing is fundamentally impossible in CPython; the registry/GC-cooperation half is genuinely EASIER (deterministic refcounting + weakref callbacks beat ReferenceQueue polling), while the genuinely harder parts are orthogonal to GC: weakref-ability of builtins (a wrapping/design decision) and the absence of a bounded heap for quota semantics (use RSS budget). The 440-line /Users/sh/pyrsistance/resources/store/storage/storage/GC.md is the single best design document to mine for the storage-side GC and should be ported as the spec for pyrsistance's disk-GC/registry contract.
