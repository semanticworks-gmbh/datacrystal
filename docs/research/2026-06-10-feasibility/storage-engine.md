# Finding: storage-engine

_Feasibility study, 2026-06-10. One of 10 parallel investigations (5 reading the EclipseStore source in `resources/store`, 5 researching the Python ecosystem)._

## Summary

EclipseStore's embedded storage engine (resources/store/storage/storage/) is a per-channel, append-only, log-structured store. N channels (power of 2, default 1) each own a dedicated worker thread, a slice of the OID space (channelIndex = objectId & (channelCount-1)), their own directory of data files, their own transaction log, and an in-RAM entity index/cache. Stores are two-phase: each channel appends its chunk to its head file and logs a Store entry; a barrier task commits all channels together (commit = just advancing the in-memory committed-length counter; crash recovery truncates uncommitted tails). Startup reads every data file once to rebuild the per-OID index in RAM (newest version wins, scanning files in reverse), validates file lengths against the transaction log, truncates torn writes, then loads the root object graph eagerly into JVM objects (Lazy<> references being the only on-demand boundary). Housekeeping runs incrementally inside each channel thread under nanosecond time budgets: storage-level mark&sweep GC coordinated across channels via mark queues, entity-cache eviction, and file "dissolving" (copy live data forward, delete file) when a file's payload ratio drops below 0.75 or it is <1MB/>8MB. Crash consistency is achieved by append-only writes + transaction-log-before-action ordering + truncate-on-boot; notably, there is no fsync/FileChannel.force anywhere in the NIO AFS layer, so durability is delegated to the OS page cache.

## Key findings

### Channel architecture: fixed power-of-2 partitions, one thread + one file set + one entity index each

StorageChannel.Default (storage/storage/.../types/StorageChannel.java) is a Runnable worker owning a StorageFileManager.Default and StorageEntityCache.Default. Channel count is fixed at startup by StorageChannelCountProvider (default 1, must be 2^n, max 1024) and partitions entities by OID: StorageEntityCache.oidChannelIndex = (int)(objectId & (channelCount-1)). The channel run() loop pulls tasks from a shared StorageTaskBroker (tasks form a linked list; every channel processes every task on its own slice), and interleaves housekeeping (file cleanup, GC, cache check, transaction-log compaction) in idle gaps under a budget (default: 10ms budget per 1000ms interval, StorageHousekeepingController.Defaults). Cross-channel operations synchronize via StorageChannelSynchronizingTask.AbstractCompletingTask: waitOnProcessing() barrier, then succeed/fail per channel.

### On-disk format: per-channel append-only data files + binary transaction log

Layout (StorageFileNameProvider.Defaults): channel_N/channel_N_M.dat data files, channel_N/transactions_N.sft log, a shared PersistenceTypeDictionary.ptd (text) and used.lock. Data files are pure concatenations of serialized entities; each entity record is [8B length][8B typeId][8B objectId][field data] in native byte order (Binary.getEntity*RawValue at fixed offsets; 24-byte header). Negative length = comment/gap to skip. Files roll over at fileMaximumSize (default 8MB; also hard 2^31 limit because positions are ints). The transaction log has 5 fixed-size little-structured entry types (StorageTransactionsAnalysis.Logic): FILE_CREATION(26B), STORE(18B), TRANSFER(34B), FILE_TRUNCATION(34B), FILE_DELETION(26B), each [1B length][1B type][8B timestamp][8B resulting file length]{8B file number}{8B special offset}. The log records file-length checkpoints, not per-entity data.

### Commit protocol: append + log entry per channel, then barrier commit; commit itself is just a length counter

storer.commit() on the app side sends one Binary chunk set to StorageRequestTaskStoreEntities. Per channel: storeChunks() appends buffers to the head file, computes storagePositions, writes a STORE entry to the transaction log, sets uncommittedDataLength. After ALL channels report success at the task barrier, each calls commitChunkStorage() -> fileManager.commitWrite() which merely does headFile.increaseContentLength(uncommittedDataLength) — an in-memory acknowledgment. fail() -> rollbackChunkStorage() physically truncates the head file back. Afterwards postStoreUpdateEntityCache() registers/updates each written entity in the in-RAM index (putEntity + markEntityForChangedData for GC correctness + file.appendEntry). The store timestamp is shared across channels and is the consistency token used at recovery.

### Store semantics: identity-based OID registry; lazy storer writes explicit arg + new objects only; eager rewrites everything

Documented in docs/modules/storage/pages/storing-data/lazy-eager-full.adoc (serializer internals live in the eclipse-serializer repo, not this clone). Every persisted object gets an objectId in a PersistenceObjectRegistry keyed by Java object identity (==, not equals). storer.store(x): x is ALWAYS re-serialized; the graph walk then skips any referenced child already in the registry (lazy, the default for all EmbeddedStorageManager.store/storeAll/storeRoot convenience methods) or re-writes every reachable object (eager storer, createEagerStorer()). No dirty tracking exists — 'the object that was modified must be stored' is an application obligation; in-place mutation of an already-registered child is silently not persisted under lazy. Per-field eager override via PersistenceEagerStoringFieldEvaluator. Data is buffered in the storer until commit(); no commit = no effect.

### Housekeeping GC: incremental tri-color mark & sweep over storage entities, coordinated across channels

StorageEntityCache + StorageEntityMarkMonitor implement mark&sweep over the persistent graph (independent of JVM GC). Each channel has a StorageObjectIdMarkQueue; a StorageReferenceMarker routes discovered reference OIDs to the owning channel's queue (objectId & channelHash); a central markMonitor tracks pendingMarksCount and per-channel needsSweep flags, with hot/cold GC phases. Entities are marked gray/black; stores during marking demote affected entities to gray and re-enqueue ('doomed kept alive'/'slipped through' handling in markEntityForChangedData). Sweep deletes white entities UNLESS still reachable in the application's object registry (LiveObjectIdsHandler predicate guards against collecting objects alive in JVM memory). Deletion = removal from the in-RAM index only; the bytes become 'gaps' reclaimed later by file cleanup. All steps run incrementally under nano-time budgets inside the channel thread.

### File dissolving driven by live-data ratio: copy-forward then delete, logged as TRANSFER

StorageDataFileEvaluator.needsDissolving (defaults: fileMinimumSize 1MB, fileMaximumSize 8MB, minimumUseRatio 0.75): a file is dissolved if dataFillRatio (live bytes / total bytes) < 0.75, or it is under-sized or over-sized (except gapless single-entity files). StorageFileManager.incrementalDissolveStorageFile/transferOneChainToHeadFile copies contiguous live-entity chains from the victim file to the current head file via file-to-file transfer (writer.writeTransfer), updates each entity's storagePosition in RAM, writes a TRANSFER log entry, and when the victim has no live content left, writes FILE_DELETION to the log BEFORE physically deleting (explicit comment at StorageFileManager.deleteFile explains the ordering invariant). A round-robin fileCleanupCursor walks the circular doubly-linked file list under time budget; entity-cache eviction is separate (StorageEntityCacheEvaluator, default timeout 24h, threshold ~1GB, age*size weight function).

### Startup: full scan rebuilds per-OID index in RAM; object graph loaded eagerly except Lazy<>

StorageChannelTaskInitialize: each channel reads its inventory (data files + parsed transaction log), a consistent store timestamp is agreed across channels (headFileLatestTimestamp of channel 0, falling back to headFileLastConsistentStoreTimestamp; mismatch -> StorageExceptionConsistency). StorageFileManager.initializeStorage validates every file's physical length against its log entry (head file may be longer = torn write), then StorageEntityInitializer reads ALL data files newest-to-oldest into a buffer, indexes entity offsets, and registers only the FIRST occurrence of each OID (= latest version); older versions become gaps. So boot cost is one full read of the store and the index (StorageEntity.Default per OID: position, length, type, GC state, cache ptr) is fully RAM-resident. The binary payload is optionally kept in the off-heap cache (initiallyCacheEntity). Then EmbeddedStorageManager.start() -> initialize() loads PersistenceRoots via loader.loadRoots() and synchronizes them with the application-defined root; building the root object materializes the entire non-Lazy reachable graph into JVM objects (docs/loading-data: 'all objects ... loaded immediately'); Lazy<T> holders stop traversal and reload via OID on get(), with LazyReferenceManager handling timeout-based unloading.

### Crash consistency: append-only + log-before-action + truncate-on-recovery; NO fsync

Guarantees come from ordering, not flushing: (1) stores append to one head file only and write a STORE log entry; the in-memory commit happens after an all-channel barrier; (2) on boot, any bytes in the head file beyond the last logged consistent length are truncated (handleLastFile writes a FILE_TRUNCATION entry before truncating; optional rescue copy via provideTruncationTargetFile, .bak suffix); (3) file deletions are logged before physical deletion so a crash in between is recoverable; (4) a transfer that completed without its log entry is harmless — the source file can't have been deleted yet, so its data is re-registered on boot (explicit design comment in appendBytesToHeadFile); (5) cross-channel atomicity via the shared store timestamp: a channel whose last store wasn't logged everywhere falls back to the prior consistent timestamp and truncates. CAVEAT: grep shows no FileChannel.force()/fsync in afs/nio (or the writer); writes go to the OS page cache, so a power failure (vs process kill) can lose recent commits — EclipseStore is consistent-but-not-durable-to-device by default. StorageLockFileManager (used.lock) prevents two processes opening one storage; full continuous backup (StorageBackupHandler) mirrors data+log writes to a backup directory.

## Implications for the Python port

Replicate: (1) the append-only data file + tiny fixed-size transaction-log design — it is simple, debuggable, and gives crash consistency with ~40 lines of recovery logic (validate lengths against log, truncate torn tail, replay nothing); (2) the in-RAM entity index {oid -> (file, offset, length, typeId)} rebuilt by a single reverse-order scan at boot with newest-version-wins — this is the heart of the engine and is straightforward in Python (dict[int, EntityRef]); (3) the storer semantics: identity-based object registry (id(obj)/weakref -> OID), lazy walk (skip registered children) as default, eager walk as opt-in, buffer-until-commit; mirror the 24-byte [length][typeId][objectId] record header (struct '<qqq' or '=qqq'); (4) log-entry-before-destructive-action ordering for truncate/delete; (5) a Lazy[T] proxy/descriptor as the on-demand loading boundary. Simplify: start with ONE channel — the entire channel apparatus (OID hashing, per-channel threads, mark queues, barrier tasks, cross-channel consistent timestamps) exists for Java multi-core write parallelism that a GIL-bound Python process cannot exploit; with one channel, store-commit degenerates to append+log+counter and initialization needs no cross-channel timestamp negotiation, while the file format stays forward-compatible with multi-channel later (keep channelIndex in file names and the & (n-1) hash so n=1 is the trivial case). Replace the incremental tri-color storage GC + LiveObjectIdsHandler machinery (the single most complex subsystem, ~2000 lines across StorageEntityCache/StorageEntityMarkMonitor) with a stop-the-world or generational offline mark&sweep run explicitly or at idle — acceptable because its only job is reclaiming disk gaps, and file dissolving by live-ratio (keep the 0.75 default, 1MB/8MB bounds) can run as a simple background task or even on-demand. Skip the off-heap entity binary cache entirely at first (Python objects ARE the cache; mmap the data files for reads instead). Do differently: add explicit fsync (configurable: per-commit / interval / never) since EclipseStore's silent no-fsync is a real durability gap; consider making dirty-tracking first-class (e.g. pydantic/dataclass __setattr__ hooks) to remove the notorious 'lazy storer misses in-place child mutations' footgun, which the Java side cannot fix for arbitrary objects but Python instrumentation can; and keep eager-load-at-boot but make Lazy the default for large collections (their GigaMap lesson). Be aware loading requires the type dictionary (typeId -> class/field layout) — persist it as a sidecar text/JSON file like PersistenceTypeDictionary.ptd to support schema evolution (legacy type mapping) later.
