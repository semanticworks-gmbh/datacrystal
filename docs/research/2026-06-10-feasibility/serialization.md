# Finding: serialization

_Feasibility study, 2026-06-10. One of 10 parallel investigations (5 reading the EclipseStore source in `resources/store`, 5 researching the Python ecosystem)._

## Summary

EclipseStore's persistence layer (Eclipse Serializer) converts arbitrary Java object graphs to a flat sequence of binary entity records, each with a 24-byte header [8B length, 8B typeId, 8B objectId]; references are stored as 8-byte OIDs, primitives inline at natural width. Structure metadata lives NOT in the binary records but in a human-readable text file, PersistenceTypeDictionary.ptd ("schema-as-data"): one entry per (typeId, class-structure) version, appended-to whenever a class changes. Schema evolution is handled at load time via Legacy Type Mapping: the startup diff between stored type dictionary and current runtime classes triggers heuristic (Levenshtein) field matching, optionally overridden by an explicit refactorings.csv (map/delete/add semantics) or a custom BinaryLegacyTypeHandler; stored data is never rewritten, only re-encoded in the new format when re-stored. Security: deserialization never executes user code (no readObject/constructors; instances allocated via Unsafe and fields set directly), eliminating gadget-chain RCE by design (validated by an NCC Group review). For Python: pickle matches the graph-handling power but fails on exactly the two pillars (security, explicit schema evolution); msgspec/cattrs are safe and typed but tree-shaped with no shared-reference identity; the right analog is a custom binary format with OID swizzling + a type-dictionary file, using dataclass/pydantic introspection as the type-handler generator (ZODB is the proven prior art for the OID/lazy-ghost side).

## Key findings

### Type handlers: reflection-generated, per-class binary codecs

Each class gets a BinaryTypeHandler generated automatically by runtime reflection — no Serializable, no annotations, no default constructor required. Handlers implement store(), create() (instance allocation WITHOUT constructor), updateState() (set fields from binary), and iterateLoadableReferences() (enumerate contained OIDs so the loader can fetch them). A registry maps Class->handler; unknown types are auto-registered on first serialization, but explicit registration (SerializerFoundation.registerEntityTypes) is recommended for stable type IDs. Excluded: JVM-bound types (Thread, IO streams) and lambdas/anonymous classes (unstable synthetic names). Custom handlers: extend CustomBinaryHandler, or a zero-boilerplate static provideTypeHandler() method on the entity class returning Binary.TypeHandler(...) with field getter/setter lambdas (docs: resources/store/docs/modules/storage/pages/customizing/custom-type-handler.adoc, serializer/pages/custom-type-handlers.adoc).

### PersistenceTypeDictionary.ptd: human-readable schema-as-data, append-only versioning

A plain-text file in the storage base dir (default name PersistenceTypeDictionary.ptd) is the single source of structural truth; binary records carry only typeIds. Format (confirmed against a real .ptd from a live store): 19-digit zero-padded typeId + space + FQCN + '{...}'; members one per line ending ','; e.g. '0000000000000000030 java.lang.String{ [char] value, }'. Tokens: 'primitive 8 bit integer signed' for primitive defs, '[char]'/'[byte]' for inlined var-length data, '[list] name( elemType element, )' for nested complex lists, 'Owner#field' qualifier for inherited/shadowed fields. Built-in TIDs: 1-8 primitives, 10 Object, 30 String, 43 ArrayList; user/runtime types start at TID 1_000_000 (Persistence.START_TID_REAL). When a class's structure changes, a NEW entry with a NEW typeId is appended — multiple structural versions of the same class name coexist, which is what makes lazy on-load migration possible. The serializer can export/import the dictionary as a String (exportTypeDictionary / setInitialTypeDictionary), and TypedSerializer can embed it in the payload with strategies TypeDictionary/Diff/IncrementalDiff.

### ID space: one 64-bit long partitioned into TID/OID/CID ranges

From org.eclipse.serializer.persistence.types.Persistence: TIDs occupy [1, 1e18) with real types from 1_000_000; OIDs occupy (1e18, 9e18) (START_OID_BASE=1_000_000_000_000_000_000); constant-IDs (CIDs, for JVM-global constants like enum instances so identity survives) occupy (9e18, 9.1e18) with user constants from 9e18+10_000. Because ranges are disjoint, any 8-byte reference slot is self-classifying. java.lang.Class is persisted as its TID in a long field. The root is registered under identifier "ROOT" via PersistenceRoots$Default (itself an ordinary entity in the dictionary, TID ~1000061 in practice).

### Binary graph encoding: flat entity records, references as OIDs, list headers vs array bombs

From org.eclipse.serializer.persistence.binary.types.Binary: every entity = 24-byte header [length(8) at offset 0, typeId(8) at 8, objectId(8) at 16] + content at offset 24. References are 8-byte OIDs inline; primitives inline (1/2/4/8 bytes); variable-size content uses a 16-byte list header [byteLength(8), elementCount(8)] that is cross-validated to defend against 'array bombs' (forged element counts). A graph is a sequence of such independent records; cycles and shared references are free because edges are OIDs. Loading: read header -> handler lookup by TID -> create() without constructor -> iterateLoadableReferences collects OIDs to fetch -> updateState resolves OIDs via handler.lookupObject (object registry with weak refs preserves reference identity, restored.orders.get(0).customer == restored). Lazy<T> wraps just an OID and deserializes on first get().

### Legacy Type Mapping: load-time transformation, never data migration

On startup EclipseStore diffs the stored type dictionary against current classes; for each changed type it runs: analysis -> matching -> confirmation -> compile value translators -> per-load translation (one-time cost at init; legacy loads then run at full speed). Stored records are untouched; a record is rewritten in current format only when its object is naturally re-stored ('over time, all legacy records migrate'). Auto-handled with zero config: field added (default null/0/false), removed (discarded), reordered, renamed (Levenshtein heuristic over name similarity + type compatibility, scores shown like 'name -0.750 ----> lastname'), primitive widening/narrowing, primitive<->wrapper boxing (null->0). Requires explicit mapping: class rename/package move, ambiguous renames, rename+retype. Source: resources/store/docs/modules/storage/pages/legacy-type-mapping/index.adoc.

### The refactoring mapping file (refactorings.csv): three-operation DSL

Registered via foundation.setRefactoringMappingProvider(Persistence.RefactoringMapping(path)). Semicolon/tab-separated, columns 'old;current'. Three ops: MAP 'old;new' (rename), DELETE 'old;' (discard data), ADD ';new' (declare field as genuinely new so the heuristic won't mis-bind it). Syntax: fields 'pkg.Class#field', classes 'pkg.OldClass;pkg.NewClass' (field heuristic still applies inside), inheritance disambiguation 'OwningClass#DeclaringClass#field', and version pinning by typeId '1012345:pkg.Class;pkg.Class' when multiple stored versions exist. Explicit and heuristic mappings mix — only specify what the heuristic gets wrong. A class with no current mapping ('old;') becomes a PersistenceUnreachableTypeHandler (class can be deleted from codebase; throws only if an instance is actually loaded). The confirmation step is pluggable via PersistenceLegacyTypeMappingResultor (default: LoggingLegacyTypeMappingResultor accepts all + logs); heuristic pluggable via PersistenceMemberSimilator (enables e.g. @MappedFrom annotation matching).

### Custom legacy type handlers for structural migrations

When field mapping can't express the change (split one field into an object, computed values), a BinaryLegacyTypeHandler.AbstractCustom<T> declares the OLD binary layout (CustomField list + manual byte offsets, e.g. BINARY_OFFSET_directions = name_offset + Binary.objectIdByteLength()), reads old OIDs/values in updateState, and constructs the NEW object shape. It must also implement iterateLoadableReferences so the loader pre-fetches referenced entities. Registered via connectionFoundation.getCustomTypeHandlerRegistry().registerLegacyTypeHandler(...). Full worked example (NicePlace.directions -> Location object) in resources/store/docs/modules/storage/pages/legacy-type-mapping/index.adoc lines 417-539.

### Security stance: data-only deserialization, no code execution, by construction

Eclipse Serializer is a from-scratch format, unrelated to java.io.Serializable: no readObject/writeObject callbacks, no constructor invocation (Unsafe.allocateInstance + direct field writes via XMemory), handlers move only data. Therefore ysoserial-style gadget-chain RCE is impossible by design; marketing/docs cite a 2021 NCC Group technical review by Robert Seacord confirming the design (microstream.one blog). The type registry acts as an implicit whitelist: the default Serializer carries no type info at all (receiver must pre-register types), and only dictionary-registered, handler-backed types can be instantiated. Remaining caveats EclipseStore acknowledges or implies: no built-in auth/encryption (FAQ: delegated to the application; storage files are trusted input), constructor bypass means class invariants are NOT re-validated on load, and TypedSerializer's embedded dictionary widens the attack surface slightly (new structure registration, still no code exec).

### Python analog assessment: pickle/ZODB vs msgspec/cattrs vs custom format

pickle: only stdlib option that natively handles cycles + shared refs (memo table) and arbitrary classes, and its persistent_id/persistent_load hooks are precisely EclipseStore's OID-swizzling extension point (ZODB is built on exactly this: _p_oid, ghost objects = Lazy refs, per-object pickles in FileStorage). But it fails both EclipseStore pillars: load-time code execution via __reduce__ (RCE on untrusted bytes; even restricted Unpicklers are whitelist hacks) and implicit schema (class import path + __dict__ snapshot: renamed class = ImportError, renamed field = silently stale attribute, no type dictionary, no migration hook beyond __setstate__). msgspec: extremely fast, schema'd Structs, safe (no code exec), but strictly tree-shaped — no shared-reference identity, no cycles, no per-object records; ideal as the leaf-record VALUE codec, wrong as the graph encoder. cattrs: flexible un/structure hooks for attrs/dataclasses, same tree-shape limitation, slower. Conclusion: none is sufficient alone; a custom entity-record binary format with an OID registry and a type-dictionary file, using msgspec/struct for the inner payload encoding, is the honest analog.

### Dataclasses/pydantic as the typed schema layer with explicit evolution

Dataclass/pydantic introspection (dataclasses.fields() / model_fields with type annotations) plays exactly the role of Java reflection in EclipseStore's automatic type-handler generation: field names + annotated types = the type dictionary entry. Constructor bypass exists in Python: object.__new__(cls) + __dict__/__slots__ writes, and pydantic's model_construct() (skips validators) — the analog of Unsafe.allocateInstance. Evolution design: keep an append-only, human-readable type dictionary (JSON or .ptd-like text) with monotonic TIDs; on startup diff stored entries vs current models; auto-handle add/remove/reorder/widen; difflib.SequenceMatcher or Levenshtein for rename heuristics; explicit mapping via a refactoring file (CSV/TOML) AND/OR inline markers (Annotated[str, RenamedFrom('name')], a default_factory for added fields) which Python can do more ergonomically than Java's CSV. Restricting the persistable universe to dataclasses/pydantic/attrs + builtin containers + bytes/numpy/arrow buffers (rather than EclipseStore's 'any POJO') is both simpler and the security win: the type dictionary + registered handlers become a hard whitelist, and no callable is ever evaluated at load time.

## Implications for the Python port

Replicate: (1) the three-part architecture — per-type handlers generated from dataclass/pydantic introspection, a 64-bit OID swizzling registry (id-keyed strong map during a store transaction + weakref OID->object map for load identity; beware Python id() reuse, so key on the registry's own holds, not raw id() longevity), and an append-only human-readable type dictionary file whose TIDs are the only type metadata in binary records. (2) The entity record format [length, tid, oid] + refs-as-OIDs + inline scalars + length-and-count list headers (validate both — the 'array bomb' defense). Keep the partitioned long ID space (TID < OID < CID ranges) so any 8-byte slot is self-classifying; use the CID range for enum members and interned sentinels so identity survives restarts. (3) Legacy Type Mapping verbatim: startup diff of dictionary vs current models, new TID per structural version, compiled per-(legacyTID->class) converter functions, auto rules (add/remove/reorder/widen/Optional-boxing), rename heuristic with an overriding explicit mapping file, an 'unreachable type' tombstone for deleted classes, and a pluggable resultor hook (great place for an interactive 'confirm mapping' dev UX that EclipseStore only half-has). (4) The security posture as a headline feature vs pickle: instantiate via object.__new__/model_construct, set fields directly, never call user callables at load; only dictionary-registered types are instantiable. Simplify/differ: skip storage channels and the AFS abstraction initially (single file or directory of segment files + a transaction log is enough); restrict persistable types to dataclasses/pydantic/attrs + builtins + bytes/numpy/pyarrow buffers instead of 'any object' — this dodges Python's messy reflection (C-extension types, __slots__ variance) and strengthens the whitelist; prefer Annotated field markers (RenamedFrom, Discarded, default_factory) as the primary evolution API with the CSV/TOML refactoring file as the escape hatch for class renames and cross-version pins; use msgspec or struct for inner payload encoding rather than hand-rolled byte twiddling; for pydantic, run validation optionally on load (EclipseStore can't re-validate invariants — Python can offer it as a flag, a genuine improvement). Study ZODB before building: it proves the OID/ghost(Lazy)/per-object-record model works in Python, and its pain points (pickle security, zope.generations migration scripts, no type dictionary) are precisely the gaps this design fills. Lazy references should be a descriptor/proxy holding an OID, mirroring Lazy<T>/ZODB ghosts, and the GraphQL/FastAPI layer can treat OIDs as node IDs directly.
